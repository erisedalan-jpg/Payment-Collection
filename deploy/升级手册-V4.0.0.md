# 升级手册 —— V4.0.0（蓝信推送集成；从 V3.5.0 升级）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`）的 **V3.5.0** 升级到 **V4.0.0**。
> 本包是「**非纯前端**」增量包——**换 dist + 覆盖后端 + 重启后端**。
> 随附最小更新包：`pmplatform-update-V4.0.0.zip`。
> 若现网早于 V3.5.0，请先用对应旧手册升到 V3.5.0，再用本包。

---

## 0. 本次升级包含什么（V3.5.0 → V4.0.0）

**新增「蓝信推送」能力**：管理员在 `/data`「配置」签一键把**项目关注原因**与**倚天工时填报问题**，经蓝信应用号消息推送给相关人员。**手动触发 + 发前预览**，绝不自动发。

这是平台第一次具备「**主动对外发消息、且触达真实员工**」的能力面，故定为 X 级大版本。

**它做什么**
- **工时问题** → 推给**填报人本人**
- **项目关注原因** → 推给**项目经理本人**
- **汇总** → 推给上级（**可配 0..5 级累积**：0=不发；1=直接上级；2=+隔级；3=部门级；4/5 预留）
- 三类卡片均为蓝信 `appCard`，汇总卡按「直接下属 × 原因」嵌套聚合、逐层卷上去

**超管可配**（`/data`「配置」签 →「蓝信推送」卡）
- 总开关、凭证（AppId / AppSecret / 组织ID / 网关地址）
- 每条路由的启停 + **逐项勾选**（哪些问题码、哪些关注原因参与推送）
- 是否发给本人／项目经理、汇总到上级第几级
- **连通性自检**：取 AppToken → 用测试工号换蓝信人员ID → 给该工号本人发一条测试消息。**全程不触碰他人**

**安全设计**
- **发前预览**：先看「要发给谁、每张卡片长什么样」，确认后才真发；预览与实发走同一份计算，**所见即所发**
- **二次确认**：点「确认推送」会弹窗提示「会真实触达员工，不可撤销」
- **并发锁**：同一时刻只允许一次推送，重复点击直接拒绝（推送不可撤销，双击就是全员收两条）
- **不静默丢弃**：未解析的收件人、发送失败的条目，都在界面上显式列出
- **密钥不回显**：AppSecret 存服务器本地、不回传前端、不进日志与审计

> **【头号注意 1】⚠ 本包需要重启后端。** 新增 3 个后端模块（`lanxin.py` / `lanxin_config.py` / `lanxin_recipients.py`）并改动了 `server.py` / `audit.py`，不重启则新端点不存在、页面上的蓝信卡会报错。
>
> **【头号注意 2】✅ 无需点「更新数据」。** 蓝信推送不进数据管线，配置改完即时生效。
>
> **【头号注意 3】✅ 无新增页面 / 路由 / pageKey / 授权。** 配置卡与推送抽屉都在既有的 `/data`「配置」签下、**超管专属**，现有账号授权一律不用动。
>
> **【头号注意 4】⚠ 蓝信凭证尚未申请，升级后功能处于「未启用」状态。** 这是**预期的**：总开关默认关闭、凭证为空，此时点推送会被拒绝并提示「蓝信推送未启用」。等凭证到位后按 §4 启用即可。**升级本身不受影响，其余所有功能照常。**

## 2. 升级步骤（Ubuntu 服务器，约 2 分钟）

```bash
# 0) 上传 pmplatform-update-V4.0.0.zip 到服务器任意临时目录并解压
cd /tmp && unzip -o pmplatform-update-V4.0.0.zip

# 1) 备份(建议:本次动后端,回滚要用)
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$(date +%Y%m%d-%H%M)
sudo mkdir -p /opt/pmplatform/bak-$(date +%Y%m%d-%H%M) && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$(date +%Y%m%d-%H%M)/

# 2) 覆盖后端 .py(核心:含 3 个新模块)
cd /tmp/pmplatform-update-V4.0.0
sudo cp -f *.py /opt/pmplatform/

# 3) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 4) 重启后端(必须 —— 新端点靠这一步生效)
sudo systemctl restart pmplatform      # 若服务名不同,按现网实际名称
sudo systemctl status pmplatform --no-pager | head -5

# 5) 无需点「更新数据」。浏览器强刷(Ctrl+F5),确认左上角版本号 V4.0.0。
```

> 注：若 nginx 缓存了静态资源，强刷仍是旧版时 `sudo systemctl reload nginx` 后再刷。

## 3. 升级后验证清单（凭证到位前能验到这里）

用**超管账号**进 `/data`：

- [ ] 左上角版本号显示 **V4.0.0**
- [ ] 原有功能全部正常（三签、更新数据、上传、标签、门户、历史回滚等）—— 本次未改动它们
- [ ] 「配置」签下出现「**蓝信推送**」卡：总开关（默认关）、凭证四项输入框、两条路由（倚天工时问题 / 项目关注原因）
- [ ] 每条路由可见：启停开关、「发给本人/项目经理」勾选、汇总级别下拉（**0..5 六档**）、参与推送的项目逐项复选框
- [ ] AppSecret 框显示「**未配置**」
- [ ] 点「预览并推送」→ 抽屉打开 → 列出收件人与**每张卡片的完整文案**
- [ ] 点「确认推送」→ 二次确认弹窗 → 确认后应被拒绝并提示「**蓝信推送未启用**」（因总开关关闭）——**这是正确行为**
- [ ] 深/浅两个主题正常；console 无报错

> **关于「未解析 0 项」**：预览里「未解析」显示 0 是**正确**的，不是解析失灵。主域项目的定义就是「项目经理在组织架构花名册内」，所以经理一律解析得到。

## 4. 凭证到位后如何启用（等蓝信申请下来再做）

申请清单见 `docs/2026-07-17-蓝信开放平台接入申请清单.md`（**前置卡点：需组织管理员开通「组织开发者」身份**）。

拿到凭证后，**全程在页面上操作、无需改代码、无需再升级**：

1. `/data` →「配置」签 →「蓝信推送」卡，填入 **AppId / AppSecret / 组织ID / 网关地址**，点「保存配置」。
2. 在「自检工号」填**你自己的工号**（如 `A000701`），点「**连通性自检**」。三步全绿即接入完成：

| 步骤 | 失败意味着什么 | 怎么办 |
|---|---|---|
| ① 取应用访问TOKEN | AppId / AppSecret / 网关地址有误 | 回 §4.1 核对，改配置即可 |
| ② 工号换人员ID | 组织ID 有误，**或蓝信「人员编号」≠ 我方工号** | 先核组织ID；若无误即编码不一致 → 见下方 ⚠ |
| ③ 发测试消息给本人 | 消息接口权限未开通（错误码 `10005`） | 请蓝信组织管理员开通 `messages/create` |

3. 三步全绿后，打开「总开关」，再点「预览并推送」——**先看清收件清单与卡片文案**，确认无误再点「确认推送」。

> **⚠ 若自检第②步失败且组织ID无误**：说明蓝信通讯录的「人员编号」与我方 `组织架构.xlsx` 的「工号」不是同一套编码。
> 应对：由贵司提供全员「工号 ↔ 人员编号」对照，我方更新一版 `组织架构.xlsx` 增加「人员编号」列，并发布 **V4.0.1** 解决。
> **注意**：改用手机号/邮箱匹配**不可行** —— 现有 `组织架构.xlsx` 不含这两列。

**建议的首次推送节奏**：先把两条路由的「汇总级别」都设为 **0（不发汇总）**、只发给本人，小范围跑通、看看卡片实际观感与同事反馈，再逐步放开到上级。

## 5. 回滚

本次动了后端，回滚需同时还原 `.py` 与 `dist` 并重启：

```bash
sudo cp -f /opt/pmplatform/bak-<时间戳>/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-<时间戳> /opt/pmplatform/frontend/dist
sudo systemctl restart pmplatform
```

> 说明：本次**未改数据管线、未改数据格式**，`data/*.json` 与 `input/` 一律不受影响，回滚无数据风险。
> `data/lanxin_config.json`（蓝信配置，含 AppSecret）是本版新增文件，回滚后会被旧版忽略、不影响运行；重新升级回 V4.0.0 时配置仍在。
