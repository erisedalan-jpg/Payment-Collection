# 升级手册 —— V4.0.5（蓝信双向闭环等 5 版累积；**从 V4.0.0 直升**）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`）的 **V4.0.0** 一次升到 **V4.0.5**。
> 本包**累积包含 V4.0.1 / V4.0.2 / V4.0.3 / V4.0.4 / V4.0.5 全部改动**——因为现网停在 V4.0.0，中间四版从未部署。
> 本包是「**非纯前端**」增量包——**换 dist + 覆盖后端 + 重启后端 + 点一次「更新数据」**。
> 随附最小更新包：`pmplatform-update-V4.0.5.zip`。
> 若现网早于 V4.0.0，请先用对应旧手册升到 V4.0.0，再用本包。

> ### 这一跳只有两件事绝对不能漏，漏了会「数据看起来丢了」或「新列全空」：
>
> **【头号注意 1】⚠ 升级前必须手动备份这两个文件（来自 V4.0.2 的自动迁移）：**
> ```
> /opt/pmplatform/data/temp_followup.json
> /opt/pmplatform/data/lanxin_config.json     # V4.0.0 若配过蓝信就有;没有则跳过
> ```
> 它们会在**首次读取时自动迁移**成新结构并立即落盘。**一旦落盘就无法用 V4.0.0 读取**——旧版把新结构当损坏文件、降级成空 store，**临时跟进的归档、蓝信的路由配置会看起来「丢了」**（其实还在文件里，旧版取不出）。**回滚前必须先还原这两份备份**，详见 §5。
>
> **【头号注意 2】⚠ 升级后必须点一次「更新数据」（来自 V4.0.1）。**
> `实际终验时间` 是 V4.0.1 新增的**管线回填字段**，不重跑管线则 `/projects` 与 `/projects/temp` 的「实际终验时间」列**全部为空**。**V4.0.0 那版是「无需更新数据」的，容易惯性跳过这一步。** 4.0.2~4.0.5 本身都不需要更新数据，这一步纯粹是替 4.0.1 补的。
>
> **【头号注意 3】⚠ 本版必须重启后端。** 五版累积改动了大量后端模块（`milestones.py` / `preprocess_data.py` / `schema.py` / `temp_followup.py` / `lanxin*.py` / `server.py` / `audit.py` / `yitian_*.py` 等）。
>
> **【头号注意 4】✅ 无新增普通账号授权项。** V4.0.5 新增的 `/data` →「蓝信回复」签是**超管专属**（普通管理员看不到），新增的 `/api/lanxin/callback` 是**免登录回调入口**（供蓝信服务端调用，不占用账号体系）。**现有账号授权一律不用动。**
>
> **【头号注意 5】✅ 蓝信双向功能上线 ≠ 立即可用。** 蓝信凭证至今未申请下来，**出站推送与入站回调从未联调过**。本次升级只是把代码就位；真正启用需先拿到凭证，详见 §7。

## 2. 升级步骤（Ubuntu 服务器，约 4 分钟）

```bash
# 0) 上传 pmplatform-update-V4.0.5.zip 到服务器任意临时目录并解压
cd /tmp && unzip -o pmplatform-update-V4.0.5.zip

# 1) ★ 备份两份【会被自动迁移】的数据文件(回滚必需 —— 见头号注意 1)
TS=$(date +%Y%m%d-%H%M)
sudo cp /opt/pmplatform/data/temp_followup.json /opt/pmplatform/data/temp_followup.json.bak-$TS
sudo cp /opt/pmplatform/data/lanxin_config.json /opt/pmplatform/data/lanxin_config.json.bak-$TS 2>/dev/null || true

# 2) 备份后端与 dist
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$TS
sudo mkdir -p /opt/pmplatform/bak-$TS && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$TS/

# 3) 覆盖后端 .py
cd /tmp/pmplatform-update-V4.0.5
sudo cp -f *.py /opt/pmplatform/

# 4) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 5) 重启后端(必须)
sudo systemctl restart pmplatform      # 若服务名不同,按现网实际名称
sudo systemctl status pmplatform --no-pager | head -5

# 6) 浏览器强刷(Ctrl+F5),确认左上角版本号 V4.0.5
# 7) ★ 进 /data 点一次「更新数据」,等 SSE 跑完 —— 这一步不能省(见头号注意 2)
```

> 注：若 nginx 缓存了静态资源，强刷仍是旧版时 `sudo systemctl reload nginx` 后再刷。

1. **标签筛选选项来源变了**（V4.0.1）：表头筛选只列出**当前数据里实际用过的**标签，没被任何项目使用的标签不再出现。只有 `/projects` 一页下沉，其余 5 页标签筛选仍在表格上方。

## 5. 回滚（回到 V4.0.0）

**本次回滚必须同时还原两份 data 文件**，否则临时跟进归档、蓝信配置会看起来「丢了」：

```bash
TS=<你升级时用的时间戳>
# 1) 还原后端与 dist
sudo cp -f /opt/pmplatform/bak-$TS/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-$TS /opt/pmplatform/frontend/dist

# 2) ★ 还原两份被自动迁移过的数据文件(不做这一步,V4.0.0 读不出归档/蓝信配置)
sudo cp -f /opt/pmplatform/data/temp_followup.json.bak-$TS /opt/pmplatform/data/temp_followup.json
sudo cp -f /opt/pmplatform/data/lanxin_config.json.bak-$TS /opt/pmplatform/data/lanxin_config.json 2>/dev/null || true

sudo systemctl restart pmplatform
```

> **回滚安全性说明**：
> - `progress.实际终验时间`（V4.0.1 回填）会作为多余字段留在 `analysis_data.json` 里，V4.0.0 不消费它，无害；**回滚后无需再点「更新数据」**。
> - V4.0.5 的回调若在升级期间收到过消息，会新建 `data/lanxin_inbox.json` / `data/lanxin_callback_raw.jsonl` / `data/lanxin_raw_archive/`。这些是 V4.0.0 不认识的**新增文件**，留着无害；如需干净可删。
> - `input/` 一律不受影响。

## 6. 升级后浏览器端的一次性行为（不是异常）

首次打开相关页面时，前端会做两次**一次性**的本地存储键迁移，各只做一次、之后不再覆盖：

1. **`/projects` 标签列**（V4.0.1）：把标签列并入你的默认可见列。
2. **`/projects/temp` 选列与排序**（V4.0.2）：把旧的单实例列/排序偏好复制到「默认跟进」实例的新键下。

如果你换了浏览器或清过浏览器数据，看到的是默认列——属正常，重新调整即可。

## 7. 蓝信双向功能：凭证到位后怎么启用（V4.0.5）

**当前状态：代码已就位，凭证未申请，出站入站均未联调。** 拿到凭证后：

1. 按 `docs/2026-07-20-蓝信接口与回调地址填写说明.md` 到蓝信开发者中心：开通「组织开发者」身份 → 取 AppId / AppSecret / orgId / 网关四项 → 在「事件回调」页另取**回调密钥**与**回调签名令牌**两项。
2. 在蓝信侧填**订阅事件回调地址**（`/data` 蓝信卡里有现成的「回调地址」可一键复制），并订阅事件：应用号身份订阅 `account_message`；智能机器人身份订阅 `bot_private_message` + `bot_group_message`。
3. 在 `/data` 蓝信卡填入六项凭证、跑三步连通性自检。
4. **首次推送先把两条路由的汇总级别都设为 0、只发给本人**，自己在蓝信里回一条，确认它出现在「蓝信回复」签里，再逐步放开到上级。收件人是真人，第一条发出去的观感很难重来。

> ⚠ **联调时特别留意两处**（本版的已知假设，只有真实数据能验证）：
> - **收件箱里的「未解析」条目**：蓝信文档的字段表与实际报文键名对不上，代码两套都认了，但真实报文形态至今没人见过。有未解析条目就是解析器要调整的信号，别忽略。
> - **蓝信卡上的「已拒绝」计数**：若它涨而收件箱空，看**原因**字段——`signature` 是签名令牌填错；`stale` 则很可能是我们对回调 `timestamp` 格式的假设错了（我们按 epoch 秒解读，蓝信文档未记载该字段单位）。后一种情况服务端日志里有被拒的时间戳原值。
