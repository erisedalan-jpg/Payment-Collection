# 升级手册 —— V4.4.1（修复 `/yitian/detail`「工作成果」列全空 + 含 V4.4.0 跟进表自定义列；**全量后端快照**）

> 适用：已用 /pm 部署在 Ubuntu 服务器（nginx:80 → app 127.0.0.1:8080，systemd，`/opt/pmplatform`，venv `/opt/pmplatform/.venv`）。
> 本包是**全量后端快照**（glob 打包全部根 `*.py`）——**换 dist + 覆盖后端 + 重启 app + 点一次「更新数据」**。
> 随附最小更新包：`pmplatform-update-V4.4.1.zip`。

## 0. 本次修什么

**主因：`/yitian/detail`「工作成果」列全空。**
- **根因**：生产 `input/yitian/holidays.csv`（工作日历/节假日文件）是 **GBK 编码**（中文 Excel「另存为 CSV」在中文 Windows 默认存 GBK/ANSI）；旧 `yitian_calendar.read_holidays` 只按 `utf-8-sig` 读 → `UnicodeDecodeError` → 该异常穿透 `build_yitian_data`、被 `preprocess_data.py` 的 `try/except` **静默吞掉** → 整个倚天域不重建 → `yitian_data.json` 停在旧快照（无 `ct` 字段）→ 工作成果列全空，且「更新数据」看似成功（只在进度日志留一行 `[WARN] 倚天工时域生成失败`）。
- **修法（V4.4.1）**：`read_holidays` 改「读字节 + `utf-8-sig → gb18030`（GBK 超集）回退」，两种编码都不认才降级为空日历（不阻断）。以后 Excel 存的 GBK CSV 也能读，不再连累倚天域。

> **注：若你已按「立即解封」把 `holidays.csv` 手工转成 UTF-8 并成功更新过数据**，工作成果列此刻应已恢复；本包的价值是**代码层永久兼容 GBK**，防止将来有人再用 Excel 重存该文件时复发。

**同时含 V4.4.0（若你现网还未升到 V4.4.0）**：跟进表超管自定义列——超管可为 `/projects/temp`、`/risk`、`/payment/key`、`/opportunities/key` 加「供其他管理员填写」的文本/日期列（工具栏「列设置」）。

| 内容 | 需更新数据? |
|---|---|
| 倚天工作成果列修复（`holidays.csv` 编码兼容） | **是**（要靠「更新数据」重建 `yitian_data.json` 带上 `ct`） |
| 跟进表自定义列（V4.4.0） | 否（超管即时可配） |

> ### 头号注意
>
> **① 必须点「更新数据」，并看倚天那行日志。** 覆盖后端 + 重启还不够——`yitian_data.json` 要靠「更新数据」重建才带 `ct`。跑完 SSE 在进度日志找「倚天」那行：
> - `[OK] 倚天工时域: N 行 / M 人 / ...` → 成功。
> - `[WARN] 倚天工时域生成失败: <异常>` → 仍失败，把这行异常发出来。
>
> **② 全量覆盖后端。** 用本包**全部 `*.py`** 覆盖，确保 `yitian_calendar.py`（本次修复点）与 `yitian.py`/`schema.py`/`preprocess_data.py` 同版本。
>
> **③ 既有账号/数据零影响。** 本包不含 `data/`/`input/`；账号、累积库 `yitian_store.json`、`holidays.csv` 均不动。

## 1. 升级步骤（Ubuntu，约 5 分钟）

```bash
# 0) 上传并解压
cd /tmp && unzip -o pmplatform-update-V4.4.1.zip

# 1) 备份 dist + 后端
TS=$(date +%Y%m%d-%H%M)
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$TS
sudo mkdir -p /opt/pmplatform/bak-$TS && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$TS/

# 2) 全量覆盖后端 .py
cd /tmp/pmplatform-update-V4.4.1
sudo cp -f *.py /opt/pmplatform/

# 3) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 4) 重启后端
sudo systemctl restart pmplatform
sudo systemctl status pmplatform --no-pager | head -5

# 5) ★ 网页 → 数据管理 → 更新数据,等 SSE 跑完;进度日志找「倚天工时域」那行(应 [OK])
# 6) Ctrl+F5 刷 /yitian/detail,工作成果列有内容;左上角版本号 V4.4.1
```

**更新数据后在服务器核验（用 venv python）：**
```bash
cd /opt/pmplatform && /opt/pmplatform/.venv/bin/python -c "import json;d=json.load(open('data/yitian_data.json',encoding='utf-8'));e=d.get('entries',[]);print('generatedAt',d['meta']['generatedAt'],'| 行',len(e),'| ct非空',sum(1 for x in e if x.get('ct')))"
```
`generatedAt` 变今天、`ct非空 > 0` → 成功。

## 3. 回滚

```bash
TS=<你升级时用的时间戳>
sudo cp -f /opt/pmplatform/bak-$TS/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-$TS /opt/pmplatform/frontend/dist
sudo systemctl restart pmplatform
```

> 回滚只换回 `.py` 与 dist；`data/`/`input/` 不受影响。回滚到旧后端后，若 `holidays.csv` 仍是 GBK，倚天域会再次因编码报错被静默跳过（工作成果列重新变空）——故建议保持 `holidays.csv` 为 UTF-8。
