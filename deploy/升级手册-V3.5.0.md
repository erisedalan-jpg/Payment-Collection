# 升级手册 —— V3.5.0（/data 数据管理页 Tab 化重设计；从 V3.4.0 升级）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`）的 **V3.4.0** 升级到 **V3.5.0**。
> 本包是「**纯前端**」增量包——**只换 `frontend/dist`**，后端代码本次一字未改。
> 随附最小更新包：`pmplatform-update-V3.5.0.zip`。
> 若现网早于 V3.4.0，请先用对应旧手册升到 V3.4.0，再用本包。

---

## 0. 本次升级包含什么（V3.4.0 → V3.5.0）

`/data`（数据管理）整页重设计。**功能零改动**——不新增也不删减任何能力，只重排信息架构与合并两处入口。后端零改动。

**① 三页签信息架构**
原来一页平铺六张卡（高度差 4~5 倍、自适应换列排出参差）。现改为：

- **常驻头**：状态条 + 「更新数据（重新处理）」主操作 —— **在任何签下都可见**，不随切签消失。
- **数据源** 签：项目主域 / 倚天工时域
- **配置** 签：项目标签 / 倚天合规（检查范围 + 规则配置）/ 首页门户
- **维护** 签：人工数据导入回滚 / 数据历史回滚 / **倚天累积数据管理** / 清空数据

每次进入默认落「数据源」签（不记忆上次停在哪签）。

**② PMIS 域与项目域文件合并为「项目主域」**
原来是两张并列的卡（PMIS 九表卡 / 项目域文件卡），各有各的上传框。二者其实是**同一条主域管线的输入**（PMIS 九表 + `input/` 根 CSV 都喂 `analysis_data.json`），按「文件放哪个目录」拆卡是按实现细节分类。现合并为一张卡、**一个上传框**。

**③ 上传按文件名自动分发（本次唯一的行为增强）**
把文件一股脑丢进「上传主域数据文件」，前端按文件名自动分到 PMIS / 项目域两个后端端点，你不必再分辨「这个该传哪个框」。并且：

- **认不出的文件不再静默跳过**：逐个列名给原因 —— `已跳过:xxx.xlsx（不在主域白名单）`
- **倚天文件混进主域框会被拦下并指路**：`已跳过:工时.xlsx（属倚天工时域,请在「倚天工时域」卡上传）`
- **服务端没收下也会明说**：`;失败 N 个（服务端未接收,请重试）`（旧版此处会静默少计，只显示成功数）

有跳过或失败时，提示行走黄色警示；全部成功则是中性提示。

**④ 倚天工时域卡瘦身**
`holidays.csv` 的长格式说明收进折叠；三个超管功能按性质分流 —— 合规检查范围 / 合规规则配置 → **配置**签；累积数据管理（按周删除/清空）→ **维护**签（它是破坏性数据维护，与历史回滚、清空数据同类）。

> **【头号注意 1】✅ 纯前端。无需重启后端、无需点「更新数据」、无需新增授权。** 本包只更新 `frontend/dist`，后端 `*.py` 与 V3.4.0 完全一致（包内附带的 .py 覆盖上去是同内容、无副作用）。
>
> **【头号注意 2】无新增页面 / 路由 / pageKey。** `/data` 仍是同一个页面、同一个 `pageKey: data`，三个签是页内 tab 不是子路由。现有账号授权一律不用动。
>
> **【头号注意 3】所有原有操作都还在，只是换了位置。** 若升级后有人报「找不到 XX 了」，对照 §0① 的三签清单指路即可 —— 最常见的是「清空数据」「数据历史回滚」「首页门户」，它们分别在「维护」「维护」「配置」签下。

解压出顶层目录 `pmplatform-update-V3.5.0/`，含：

- `frontend/dist/`（**已用 `--base=/pm/` 构建**）——**这是本次唯一实质变化**。
- 全部后端根 `*.py`（与 V3.4.0 一致，覆盖无副作用）、`requirements.txt`、`deploy/升级手册-V3.5.0.md`、`pmisdata/` 脚本。
- **不含** `client/`、`data/`、`input/`、`tests/`、`docs/`、`node_modules/`。

## 2. 升级步骤（Ubuntu 服务器，约 1 分钟）

```bash
# 0) 上传 pmplatform-update-V3.5.0.zip 到服务器任意临时目录并解压
cd /tmp && unzip -o pmplatform-update-V3.5.0.zip

# 1) 备份现网 dist(可选,但建议 —— 回滚就靠它)
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$(date +%Y%m%d-%H%M)

# 2) 覆盖前端 dist(核心)
cd /tmp/pmplatform-update-V3.5.0
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist
#   (可选)后端 .py 与 V3.4.0 一致,覆盖与否均可:
#   sudo cp -f *.py /opt/pmplatform/

# 3) 无需重启后端(后端未变)、无需点「更新数据」。
#    浏览器强制刷新(Ctrl+F5),确认左上角版本号 V3.5.0。
```

> 注：若你的部署用 nginx 缓存了静态资源，强刷仍是旧版时，`sudo systemctl reload nginx` 或清 nginx 缓存后再强刷。

## 3. 升级后验证清单

用**超管账号**进 `/data`：

- [ ] 左上角版本号显示 **V3.5.0**。
- [ ] 页面顶部：状态条（上次处理 / PMIS / 本机代理 / PMIS cookie / 倚天 cookie）+「更新数据（重新处理）」按钮。
- [ ] 三个签「数据源 / 配置 / 维护」齐全，默认停在「数据源」；切到「配置」「维护」时，状态条与「更新数据」按钮**仍在**。
- [ ] **数据源**签：「项目主域」卡内同时有 PMIS 九表（9 行）与项目域文件（含 `collection_stages.csv`）两组清单，文件时间正常；右侧「倚天工时域」卡有 `工时.xlsx` / `holidays.csv`，`holidays.csv 格式说明` 是可展开的折叠项。
- [ ] **配置**签：项目标签（标签库可增/改名/启停、按标签排除）、倚天合规（检查范围 + 规则配置两个折叠）、首页门户。
- [ ] **维护**签：人工数据导入回滚、数据历史回滚、倚天累积数据管理、清空数据（点「清空数据」应弹**两次**确认）。
- [ ] **点一次「更新数据（重新处理）」**：进度条走动、文案更新、完成后文件时间刷新、看板数据正常。
- [ ] **混投上传实测**（本版重点）：在「项目主域」卡一次性选中 ①一个 PMIS 九表文件 ②一个根 CSV（如 `collection_stages.csv`）③`工时.xlsx` ④一个随便改名的文件，点「上传主域数据文件」。期望：
  - 前两个上传成功，计数正确；
  - `工时.xlsx` 列入「已跳过」并提示`属倚天工时域,请在「倚天工时域」卡上传`；
  - 乱名文件列入「已跳过」并提示`不在主域白名单`；
  - 提示行是**黄色**警示（不是中性灰）。
- [ ] 深/浅两个主题切换都正常显示，窄屏（或浏览器窗口拖窄到 768px 以下）两栏自动降为单栏。

## 4. 回滚

纯前端，回滚极简：

```bash
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-<时间戳> /opt/pmplatform/frontend/dist
```

（无后端 / 数据 / 口径改动，回滚只换回旧 dist、强刷即可。）
