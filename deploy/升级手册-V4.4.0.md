# 升级手册 —— V4.4.0（跟进表超管自定义列 + **全量后端拉齐修复 `/yitian/detail` 工作成果列**；**从 V4.1.3/V4.3.1 累积升，全量后端快照**）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd，`/opt/pmplatform`）升到 **V4.4.0**。
> 本包是**全量后端快照**（glob 打包全部根 `*.py`）——**换 dist + 覆盖后端 + 重启 app + 点一次「更新数据」**。
> 随附最小更新包：`pmplatform-update-V4.4.0.zip`。

---

## 0. 本次升级为什么要做 / 包含什么

**两件事：**

1. **【本次主因】修复 `/yitian/detail` 明细页「工作成果」列全空。**
   - **根因**：线上 `data/yitian_data.json` 停在 **2026-07-17** 未再重建（`generatedAt` 未变、行数未随累积库增长、entry 无 `ct` 键）。前端「工作成果」列读 `ct` 字段，读不到 → 全空。
   - **更深一层**：点「更新数据」也没修好，是因为**线上后端 `.py` 版本不齐**（前端 dist 有「工作成果」列且 `schema.py` 要求 `ct` 字段，但 `yitian.py` 是旧的、不产出 `ct`）→ 倚天数据构建时 schema 校验抛异常，而**倚天域被 `preprocess_data.py` 的 `try/except` 静默跳过**（只在进度日志留一行 `[WARN] 倚天工时域生成失败`）→ `yitian_data.json` 原封不动。
   - **修法**：本包是**全量后端快照**，一次性把 `yitian.py`/`schema.py`/`preprocess_data.py`/`yitian_store.py` 等**拉齐到同一版本**（`yitian.py` 产出 `ct`、`schema.py` 要求 `ct`，两端一致）；覆盖后**点「更新数据」**即可用累积库（`yitian_store.json`，正文一直在库里）重建 `yitian_data.json` 并写入 `ct`，工作成果列即有内容。

2. **V4.4.0 新功能：跟进表超管自定义列。** 超管可为 4 张跟进表（`/projects/temp`、`/risk`、`/payment/key`、`/opportunities/key`）加「供其他管理员填写」的**文本/日期**列，列名 + 归档是否清空可配（`/projects/temp` 等工具栏「列设置」）。此域不进数据管线、配置即时生效。

| 域 | 动数据结构? | 需更新数据? |
|---|---|---|
| 倚天工作成果列修复 | 否（`yitian_data.json` 增 `ct`，由管线重建） | **是（本次升级的关键动作）** |
| 跟进表自定义列 | 否（`data/followup_columns.json`，超管可配） | 否 |

> ### 头号注意
>
> **① 必须点「更新数据」，且必须看倚天那一行日志。** 覆盖后端 + 重启还不够——`yitian_data.json` 要靠「更新数据」重建才会带上 `ct`。跑完 SSE **务必在进度日志里找「倚天」那行**：
> - 看到 `[OK] 倚天工时域: <行数> 行 / <人数> 人 / ... ` → 成功。
> - 若看到 `[WARN] 倚天工时域生成失败: <异常>` → 拉齐没成功，**把这行异常发出来**（可能是数据/输入问题，非版本）。
>
> **② 覆盖后端要「全量覆盖」，别只覆盖个别 .py。** 本次问题正是版本不齐（部分 .py 旧）。用本包的**全部 `*.py`** 覆盖 `/opt/pmplatform/`，确保 `yitian.py`/`schema.py`/`preprocess_data.py` 等同版本。
>
> **③ 既有账号/数据零影响。** 本包**不含** `data/`；账号、跟进记录、累积库 `yitian_store.json`（工作成果正文在此，一直都在）均不动。跟进表自定义列是超管主动配才生效、不配即老样子。

## 2. 升级步骤（Ubuntu 服务器，约 5 分钟）

```bash
# 0) 上传 pmplatform-update-V4.4.0.zip 到服务器任意临时目录并解压
cd /tmp && unzip -o pmplatform-update-V4.4.0.zip

# 1) 备份 dist + 后端(回滚用)
TS=$(date +%Y%m%d-%H%M)
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$TS
sudo mkdir -p /opt/pmplatform/bak-$TS && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$TS/

# 2) 全量覆盖后端 .py(关键:全部,拉齐版本)
cd /tmp/pmplatform-update-V4.4.0
sudo cp -f *.py /opt/pmplatform/

# 3) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 4) 重启后端
sudo systemctl restart pmplatform
sudo systemctl status pmplatform --no-pager | head -5

# 5) ★ 浏览器登录 → 数据管理 → 点「更新数据」，等 SSE 跑完
#    进度日志里找「倚天工时域」那一行:
#      [OK] 倚天工时域: N 行 / M 人 / ...   → 成功
#      [WARN] 倚天工时域生成失败: ...        → 失败,把异常发出来

# 6) Ctrl+F5 硬刷 /yitian/detail,确认「工作成果」列有内容;左上角版本号 V4.4.0
```

**更新数据后可在服务器上直接核验 `ct` 已写入：**
```bash
cd /opt/pmplatform && python3 -c "import json;d=json.load(open('data/yitian_data.json',encoding='utf-8'));e=d.get('entries',[]);print('generatedAt',d.get('meta',{}).get('generatedAt'),'| 行',len(e),'| ct非空',sum(1 for x in e if x.get('ct')))"
```
- `generatedAt` 变成今天、`ct非空` > 0 → 修复成功。
- `generatedAt` 仍是 07-17 → 倚天域仍被跳过（看步骤 5 的 `[WARN]` 异常）。

> 注：若 nginx 缓存静态资源，强刷仍旧版时 `sudo systemctl reload nginx` 后再刷。

## 5. 回滚（回到升级前）

```bash
TS=<你升级时用的时间戳>
sudo cp -f /opt/pmplatform/bak-$TS/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-$TS /opt/pmplatform/frontend/dist
sudo systemctl restart pmplatform
```

> **回滚安全性**：`data/`（含 `yitian_data.json`/`yitian_store.json`/账号/`followup_columns.json`）不被本包覆盖；回滚只换回 `.py` 与 dist。回滚后倚天工作成果列会**重新变空**（旧后端不产出 `ct`），跟进自定义列的既有配置字段被旧版忽略、无害。
