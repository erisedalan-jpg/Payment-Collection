# 升级手册 —— V4.3.1（账号权限细粒度：员工级 → 分域 → 逐页 + 配置界面重做；**从 V4.1.3 累积升，跨 V4.2.0/V4.3.0/V4.3.1 三期**）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`）的 **V4.1.3** 升到 **V4.3.1**。
> 本包是「**非纯前端**」增量包——**换 dist + 覆盖后端 + 重启 app**。**无需点「更新数据」**（权限是请求时过滤，不进数据管线）。
> 随附最小更新包：`pmplatform-update-V4.3.1.zip`（全量后端 `*.py` 快照，累积覆盖三期改动）。
> 若现网不是 V4.1.3，可直接用本包（后端是全量快照、dist 是完整构建，从任一 V4.x 升上来都成立）。

## 0. 本次升级包含什么（V4.1.3 → V4.3.1，三期累积）

| 版本 | 主要内容 | 动数据结构? | 需更新数据? |
|---|---|---|---|
| V4.2.0 | 账号数据范围**下沉到员工级**：账号加 `allowedStaff`（工号），可把可见范围配到具体员工/项目经理 | 否（仅 `accounts.json` 账号对象加字段） | 否 |
| V4.3.0 | **分数据域范围**：账号加 `domainScopes`，项目&回款/工时/商机三域各可设一个覆盖范围 | 否 | 否 |
| V4.3.1 | **逐页数据范围 + 配置界面重做**：账号加 `pageScopes`，范围解析升为**页 > 域 > 默认**三层，任意页可独立范围；`/admin` 配置界面重做（组级选页 + 默认范围 + 覆盖列表），大幅减点击 | 否 | 否 |

一句话：这三期都是**账号权限系统的细粒度增强**，全部是**请求时按账号切数据**，**不改数据管线、不动 `analysis_data.json`/`yitian_data.json` 的产出**，因此**升级后无需点「更新数据」**。

> ### 这一跳的关键点：
>
> **【头号注意 1】⚠ 必须覆盖后端 + 重启 app。** 三期改了 `auth.py`（账号模型 + 三层解析）、`server.py`（按域并集下发 + 富化 staffNames + admin 端点）、`data_scope.py`（过滤函数加工号/PM 维度）、新增 `config.py` 的 `PAGE_DOMAINS`。只换 dist、不覆盖后端并重启，权限收窄不会生效、`/admin` 新配置存不进去。
>
> **【头号注意 2】✅ 既有账号零影响、向后兼容。** 老账号的 `accounts.json` 没有 `allowedStaff`/`domainScopes`/`pageScopes` 字段 → 读取时一律视作「默认/空」→ **行为与升级前逐字一致**。新粒度是**超管在 `/admin` 主动配置后才生效**，不配就是老样子。**升级不会改动 `accounts.json`**（本包不含 `data/`）。
>
> **【头号注意 3】✅ 无需点「更新数据」。** 与 V4.1.x 那几版不同，本三期不产出新数据字段、不改 schema。**升级后不要惯性去点更新数据**（点了也无害，只是没必要）。
>
> **【知情提示】员工级配置依赖花名册。** 「可见员工」选择器与「项目经理工号匹配」读服务器上已有的 `input/组织架构.xlsx`（工号→姓名）。该文件本就是现网数据源、无需新增；缺失时选择器为空、工号级配置不可用（L4 级不受影响）。

## 2. 升级步骤（Ubuntu 服务器，约 4 分钟）

```bash
# 0) 上传 pmplatform-update-V4.3.1.zip 到服务器任意临时目录并解压
cd /tmp && unzip -o pmplatform-update-V4.3.1.zip

# 1) 备份 dist + 后端(回滚用)
TS=$(date +%Y%m%d-%H%M)
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$TS
sudo mkdir -p /opt/pmplatform/bak-$TS && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$TS/

# 2) 覆盖后端 .py(全量,含 config.py/auth.py/server.py/data_scope.py)
cd /tmp/pmplatform-update-V4.3.1
sudo cp -f *.py /opt/pmplatform/

# 3) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 4) 重启后端(必须)
sudo systemctl restart pmplatform      # 若服务名不同,按现网实际名称
sudo systemctl status pmplatform --no-pager | head -5

# 5) 浏览器强刷(Ctrl+F5),确认左上角版本号 V4.3.1
# ★ 无需点「更新数据」
```

> 注：若 nginx 缓存了静态资源，强刷仍是旧版时 `sudo systemctl reload nginx` 后再刷。

## 4. 已知的行为变化（不是缺陷）

1. **权限粒度增强**：账号可见范围从「页面 + 全局 L4」扩展为「页面 + 逐页范围（页>域>默认，可到 L4 或工号）」。**不配即老样子**。
2. **员工级隔离靠花名册**：`input/组织架构.xlsx` 提供工号→姓名；工号级/项目经理级配置依赖它。
3. **审计**：账号创建/修改的审计详情会附带「员工/分域/逐页」范围变更信息。
4. **超管专属端点** `GET /api/admin/roster`（喂「可见员工」选择器，只出工号/姓名/L4，不含隐私列）。

## 5. 回滚（回到 V4.1.3）

```bash
TS=<你升级时用的时间戳>
sudo cp -f /opt/pmplatform/bak-$TS/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-$TS /opt/pmplatform/frontend/dist
sudo systemctl restart pmplatform
```

> **回滚安全性**：`accounts.json` 里若已配过新范围字段（`allowedStaff`/`domainScopes`/`pageScopes`），回滚到 V4.1.3 后旧版**忽略这些多余字段**、按老口径（页面 + 全局 L4）工作，无害；再次升回 V4.3.1 时配置仍在。`input/`/`data/` 不受影响。回滚后**无需点更新数据**。
