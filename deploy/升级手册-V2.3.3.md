# 升级手册 —— V2.3.3（累积发布：V2.3.0 + V2.3.1 + V2.3.2 + V2.3.3 四期一并上线）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`，运行在 `.venv`）的旧版本升级到 **V2.3.3**。
> 本包是「**纯代码 + /pm 前端 dist**」的轻量增量包，**不含 data/ 与 input/**，不动服务器现有数据与账号。
> 配套部署原理见 `deploy/部署-Ubuntu24-端口80-pm.md`（首次安装看那份；本份只讲**升级**）。
> 随附最小更新包：`pmplatform-update-V2.3.3.zip`。
> **本包代码为全量覆盖所有 .py + dist**，一次升到 V2.3.3 即含 V2.3.0/V2.3.1/V2.3.2/V2.3.3 四期全部改动。现网若早于 V2.3.0（如停在 V2.2.2），被跳过版本的**首用/数据/依赖步骤**已并入本手册 §3，照做即可。

## 0. 本次升级包含什么（四期一并）

### V2.3.0（整页级）
- **新增页面 `/risk`「风险跟进」**：以风险为核心，每条风险单行（风险全列 + 项目全列 + 换行 + 可选列 + 排序）；可编辑「跟进动作 / rev结论（文本）」「下次rev时间（日期）」；超管可设范围、可「归档（留存跟进，不清空）」；历史/选列/导出同重点项目进展页。**新增页面访问 key `risk-followup`**（见 §3）。
- 首页「项目总览」与「成本分析」补「按标签排除」（继承全局排除状态）。
- 数据治理页新增「原项目数据缺失」告警。
- 「在建项目」清单：项目经理/L4组/风险/级别/项目类型/项目状态六列加排序；「关注原因」拆「总成本超支/交付成本超支」并支持按类别筛选。

### V2.3.1（子页/局部 + 修复）
- 「重点项目进展」(/projects/key) 取数口径改为 **级别 P1 或（TOP1000 且合同>100万）**——P1 项目一律入选（列表可能变多）。
- **标签丢失修复**：升级前若出现「加了标签、点更新数据后标签看似消失」，根因是前端刷新缺失（后端数据未丢），本版已修（验证见 §3）。
- 四个跟进页（重点项目进展 / 重点商机跟进 / 临时重点跟进 / 风险跟进）历史快照支持**超级管理员逐条删除**（二次确认）。
- 重点项目进展 / 临时重点跟进 / 风险跟进 客户列：售前服务类项目取原项目客户（风险跟进新增「客户」列，默认隐藏）。

### V2.3.2（口径单一来源化）
- **售前服务类项目「客户」口径统一**：后端算一次「有效客户」——售前取原项目最终客户，原项目无数据则从项目名（`售前服务-客户名-编号`）解析；非售前取本项目最终客户。全站客户列、客户筛选、**TOP1000 客户判断**统一以此为准。
- 影响：约 19 个售前项目会新进入 TOP1000（随之进入重点项目/重点商机等范围）；约 90 个原本客户列为空的售前项目，升级后客户列显示出客户名。商机页、已关闭项目页不受影响。

### V2.3.3（子页/局部）
- **三跟进页列排序**：「重点项目进展」(/projects/key)、「重点商机跟进」(/opportunities/key)、「临时重点跟进」(/projects/temp) 三页表格，**除 4 个长文本列**（本周工作进展 / 后续工作计划 / 情况备注 / 主要涉及产品）外的所有列均可**点表头排序**（含客户、项目名/商机名、金额、各比率、各日期、各级别/状态列）。
- 排序口径：合同金额、商机金额、完工%/回款率/消耗比、各日期、P1–P4 级别按数值/时间精确排序；其余文本/枚举列按字符顺序分组。
- **V2.3.3 本身是纯前端改动**（不改后端/数据），但因本包累积含 V2.3.2，整体升级仍须点「更新数据」（见下）。

> **【头号注意】本次升级必须点一次「更新数据」才生效**（因累积含 V2.3.2：改了数据契约 schema + 项目主域计算 + TOP1000 重算；前端新增字段需后端重生成 `data/analysis_data.json`）。详见 §2 步骤 (6)、§3 第 1 条。
> **无新增 Python 依赖、无 systemd/nginx 变更、无新数据源文件。**
> （若现网已经在 V2.3.2 且已点过「更新数据」，本次只补 V2.3.3 纯前端排序，理论上不必再点；但重复点一次「更新数据」无害，建议照做以确保一致。）

## 2. 升级步骤（Ubuntu /pm systemd 部署）

以部署在 `/opt/pmplatform`、运行在 `.venv`、服务用户 `pmapp` 为例。**务必先备份**。

```bash
# (1) 备份数据 + 现有代码（用于回滚）
sudo tar czf /backup/pmplatform-data-$(date +%F-%H%M).tgz -C /opt/pmplatform data
sudo tar czf /backup/pmplatform-code-$(date +%F-%H%M).tgz -C /opt/pmplatform \
  $(cd /opt/pmplatform && ls *.py) frontend/dist

# (2) 停服务
sudo systemctl stop pmplatform

# (3) 放更新包并解压（U 盘/scp 拷到服务器，例如 /tmp）
cd /tmp
unzip pmplatform-update-V2.3.3.zip      # 或 python3 -c "import zipfile;zipfile.ZipFile('pmplatform-update-V2.3.3.zip').extractall('.')"

# (4) 覆盖后端代码（*.py）
sudo cp /tmp/pmplatform-update-V2.3.3/*.py /opt/pmplatform/

# (5) 替换前端 dist（先删后拷，避免残留旧 chunk）
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r /tmp/pmplatform-update-V2.3.3/frontend/dist /opt/pmplatform/frontend/dist
# 校验是 /pm 构建（应有输出 /pm/assets/...）：
grep -o '/pm/assets[^"]*' /opt/pmplatform/frontend/dist/index.html | head -1

#   pmisdata：本版本未改下载脚本。若服务器已有 pmisdata/ 且 cookie 仍有效，跳过；
#   仅当现网尚无 pmisdata/ 时才整套拷入：
#   sudo cp -rn /tmp/pmplatform-update-V2.3.3/pmisdata /opt/pmplatform/

# (6) 归属仍归 pmapp（data/ 须 pmapp 可写）
sudo chown -R pmapp:pmapp /opt/pmplatform

# (7) 起服务
sudo systemctl start pmplatform
sudo systemctl status pmplatform        # 期望 active (running)
```

> **(8) 起服务后，登录 → 进「数据管理」→ 点一次「更新数据」**（累积含 V2.3.2 必需，见 §3 第 1 条）。等待 SSE 进度跑完、提示成功。

> **无新增 Python 依赖**：无需 pip 安装任何东西。
> **systemd/nginx 无变化**：单元与 nginx 配置不动。

## 3. 升级后必做 / 按需操作

1. **【必做】点一次「更新数据」**（累积含 V2.3.2 引入）：
   - V2.3.2 给项目主域新增了「有效客户」字段并重算 TOP1000，前端新版读这个新字段。**不点更新数据，则前端读的是旧 `data/analysis_data.json`**，表现为：售前项目客户列显示「-」、约 90 个售前客户名不出、约 19 个本应新进 TOP1000 的售前不翻「是」、重点项目/重点商机范围未纳入它们。
   - 操作：登录任一**超级管理员** → 「数据管理」→「更新数据」→ 等 SSE 跑完。无需重新上传任何文件（用服务器现有 `input/`）。
   - （V2.3.3 排序为纯前端，不依赖此步；但一次「更新数据」可一并保证 V2.3.2 口径就位。）

2. **【按需】给普通管理员授权「风险跟进」页**（V2.3.0 新增页面）：
   - V2.3.0 新增页面 `/risk`「风险跟进」，对应**新页面访问 key `risk-followup`**。超级管理员默认可见；**普通管理员需在「页面访问控制」里勾选该页**才可见。
   - 操作：超管 → 「系统管理 / 账号管理」→ 编辑相关普通管理员 → 勾选「风险跟进」页 → 保存。

3. **【验证】标签是否正常**（V2.3.1 修复点）：
   - 升级后随便给某项目加一个标签 → 点「更新数据」→ **不刷新页面**，标签应仍在（修复前需手动刷新才回来）。
   - 若**刷新页面后标签仍丢失**，说明是更深的环境问题（项目编号变动/早前升级误清 data/），请联系开发方带该环境数据排查。

4. **账号不变（不新增超管）**：本四期**未改 `auth.py`**。超管仍固定为 `admin`、`wangxutong`、`zhangyingzhe`。

## 5. 回滚

```bash
sudo systemctl stop pmplatform
# 还原代码与 dist
cd /opt/pmplatform && sudo tar xzf /backup/pmplatform-code-<时间>.tgz
sudo chown -R pmapp:pmapp /opt/pmplatform
sudo systemctl start pmplatform
```

> 回滚到旧版后，**建议再点一次「更新数据」**让 `data/analysis_data.json` 与旧版 schema 对齐（旧版前端不读 `customer` 新字段，多一个字段无副作用；但 TOP1000 会回到旧口径）。跟进/标签/商机等本地 JSON 结构四期均未破坏性变更，无需处理。普通管理员升级期间录入的风险跟进/历史删除等操作均保留在 `data/` 下。
