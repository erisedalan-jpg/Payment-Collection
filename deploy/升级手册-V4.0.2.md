# 升级手册 —— V4.0.2（临时跟进多实例 + 蓝信路由逐项拆分；从 V4.0.1 升级）

> 适用：把**已用 /pm 部署在 Ubuntu 服务器**（nginx:80 → app 127.0.0.1:8080，systemd 托管，目录 `/opt/pmplatform`）的 **V4.0.1** 升级到 **V4.0.2**。
> 本包是「**非纯前端**」增量包——**换 dist + 覆盖后端 + 重启后端**。
> 随附最小更新包：`pmplatform-update-V4.0.2.zip`。
> 若现网早于 V4.0.1，请先用对应旧手册升到 V4.0.1，再用本包。

---

## 0. 本次升级包含什么（V4.0.1 → V4.0.2）

**两件相互独立的事**：

**① `/projects/temp` 临时重点跟进 —— 多实例化**
一个跟进事项没结束就要开下一轮，原来只有一份范围+进展+归档，只能串行。现在页面顶部有**跟进事项选项卡**，每个事项各带自己的范围、当前记录、归档，可以并行跟进多轮。

**② 蓝信推送 —— 路由逐项拆分**
原来「倚天工时问题」8 个问题码共用一套收件人规则，「项目关注原因」8 类同理。现在**每一项各自配置**：是否启用 / 是否发本人 / 汇总到第几级上级。配置卡从「一组复选框 + 一个下拉」变成**逐项表格**。

> **⚠【头号注意 1】升级前请手动备份这两个文件：**
> ```
> /opt/pmplatform/data/temp_followup.json
> /opt/pmplatform/data/lanxin_config.json     # 若尚未创建则跳过
> ```
> 这两份文件会在**首次读取时自动迁移**成新结构并立即落盘。**一旦落盘就无法用 V4.0.1 读取**
> ——旧版会把新结构当损坏文件、降级成空 store，**归档会看起来「丢了」**。
> **回滚前必须先还原备份**，详见 §5。
>
> **【头号注意 2】⚠ 本版需要重启后端。** 改动了 `temp_followup.py` / `lanxin.py` /
> `lanxin_config.py` / `server.py` / `audit.py`。
>
> **【头号注意 3】✅ 无需点「更新数据」。** 两处都不进数据管线，改的是 `data/*.json`，
> 经 `/api/*` 直接读写。
>
> **【头号注意 4】✅ 收件人与卡片正文零变化。** 迁移时每一项都继承了原路由的设置，
> 所以**升级后推给谁、推哪些项与升级前完全一致**；唯一变化是汇总卡副标题统一改为
> 「**团队汇总**」，不再写具体级差，详见 §4。需要细分时再逐项调整。
>
> **【头号注意 5】✅ 无新增页面 / 路由 / pageKey / 授权项。** 现有账号授权一律不用动。

解压出顶层目录 `pmplatform-update-V4.0.2/`，含：

## 2. 升级步骤（Ubuntu 服务器，约 3 分钟）

```bash
# 0) 上传 pmplatform-update-V4.0.2.zip 并解压
cd /tmp && unzip -o pmplatform-update-V4.0.2.zip

# 1) ★ 备份两份会被自动迁移的数据文件(回滚必需)
TS=$(date +%Y%m%d-%H%M)
sudo cp /opt/pmplatform/data/temp_followup.json /opt/pmplatform/data/temp_followup.json.bak-$TS
sudo cp /opt/pmplatform/data/lanxin_config.json /opt/pmplatform/data/lanxin_config.json.bak-$TS 2>/dev/null || true

# 2) 备份后端与 dist
sudo cp -r /opt/pmplatform/frontend/dist /opt/pmplatform/frontend/dist.bak-$TS
sudo mkdir -p /opt/pmplatform/bak-$TS && sudo cp /opt/pmplatform/*.py /opt/pmplatform/bak-$TS/

# 3) 覆盖后端 .py
cd /tmp/pmplatform-update-V4.0.2
sudo cp -f *.py /opt/pmplatform/

# 4) 覆盖前端 dist
sudo rm -rf /opt/pmplatform/frontend/dist
sudo cp -r frontend/dist /opt/pmplatform/frontend/dist

# 5) 重启后端(必须)
sudo systemctl restart pmplatform      # 若服务名不同,按现网实际名称
sudo systemctl status pmplatform --no-pager | head -5

# 6) 浏览器强刷(Ctrl+F5),确认左上角版本号 V4.0.2
# 无需点「更新数据」
```

> 注：若 nginx 缓存了静态资源，强刷仍是旧版时 `sudo systemctl reload nginx` 后再刷。

## 4. 新能力怎么用

**并行跟进多个事项**：点「+ 新建」，填名称（1..20 字），范围可选「空白」或「**复制自**某个现有事项」。
你的场景是「上一轮未结束就要开下一轮」，两轮范围往往相似——**复制后微调比重配快很多**。
复制只带范围，不带进展与归档。

**蓝信逐项配置**：现在可以做到「缺少工作概述只提醒本人、工时类型填报有误要报到组长」这类区分。
汇总级别是**累积**的：填 2 表示直接上级和隔级上级都会收到。

> 汇总卡的副标题现在统一显示「**团队汇总**」，不再写「直接上级（+1）」这类具体级差——
> 因为逐项配置后，同一张卡里的行可能来自不同级别，写具体级差会自相矛盾。

## 5. 回滚

**本次回滚必须同时还原两份 data 文件**，否则归档会看起来「丢了」：

```bash
TS=<你升级时用的时间戳>
# 1) 还原后端与 dist
sudo cp -f /opt/pmplatform/bak-$TS/*.py /opt/pmplatform/
sudo rm -rf /opt/pmplatform/frontend/dist
sudo mv /opt/pmplatform/frontend/dist.bak-$TS /opt/pmplatform/frontend/dist

# 2) ★ 还原两份被自动迁移过的数据文件(不做这一步,V4.0.1 读不出归档)
sudo cp -f /opt/pmplatform/data/temp_followup.json.bak-$TS /opt/pmplatform/data/temp_followup.json
sudo cp -f /opt/pmplatform/data/lanxin_config.json.bak-$TS /opt/pmplatform/data/lanxin_config.json 2>/dev/null || true

sudo systemctl restart pmplatform
```

> **为什么必须还原数据文件**：V4.0.2 会把 `temp_followup.json` 从
> `{version, scope, current, archives}` 迁成 `{version, instances:[...]}`，把
> `lanxin_config.json` 的路由从 `issueCodes/reasons + recipients` 迁成 `items[]`。
> V4.0.1 的代码读不懂新结构，会当作损坏文件降级成空 store —— 页面上表现为
> **归档全没了、蓝信配置回到默认**。数据其实还在文件里，但旧版取不出来。
>
> 其余：`data/analysis_data.json`、`input/` 一律不受影响；本版**未改数据管线**，回滚无需重跑数据。

## 6. 升级后浏览器端的一次性行为（不是异常）

首次打开 `/projects/temp` 时，前端会把你原来存在浏览器里的**选列与排序偏好**，从旧的存储键
复制到「默认跟进」实例的新键下（因为多实例后每个事项各有一套列配置）。这个动作**只做一次**，
之后你自己调整的列配置不会再被覆盖。

如果你换了浏览器或清过浏览器数据，看到的是默认列——这属正常，重新调整即可。
