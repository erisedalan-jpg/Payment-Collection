# 蓝信回调接口调研（2026-07-20）

**结论：本轮不实现。** 经与用户确认——蓝信服务器**无法主动访问本系统**（内网部署），且当前诉求是**通知类即可、暂不做回写**。本文档归档调研事实，避免将来重新查一遍。

来源：`https://openapi.lanxin.cn/#/server-api/callback/`（注意与 V4.0.0 用的 `developer.lanxin.cn` **不是同一个站**，见 §5 抓法）

---

## 1. 回调机制确实存在，而且有「用户回复」这个事件

蓝信开放平台提供**订阅事件回调**：应用把回调地址注册到开发者中心，订阅的事件触发后，蓝信 POST 推送 JSON 过来。

**全部 14 个事件类型**：

| 事件类型 | 说明 |
|---|---|
| **`account_message`** | **人员回复应用号消息** —— 文档原话「可以支持应用和人员的一对一交互」 |
| `dept_create` / `dept_modify` / `dept_delete` | 部门增 / 改 / 删（用于组织架构同步） |
| `staff_create` / `staff_modify` / `staff_delete` | 人员增 / 改 / 删（用于组织架构同步） |
| `app_install_org` / `app_uninstall_org` | 某组织安装 / 卸载本应用 |
| `data_scope` | 应用在组织内的数据读取范围配置变更 |
| `telephone_track` | 电话追呼确认 |
| `user_logout` | 用户登出蓝信客户端 |
| `bot_private_message` / `bot_group_message` | 机器人私聊 / 群 @ 消息（需开通智能机器人能力） |

`account_message` 的载荷：

```json
{
  "from": "524288-xxx",          // 人员ID(staffId)
  "msgType": "text",             // text/image/video/file/voice/position/card/sticker
  "msgData": { "text": { "content": "this is a text", "sendTime": "1540377644020456" } }
}
```

**重要的形态差异**：这是**用户在会话里自由打字回复**，不是「卡片上点按钮」或「填表单提交」。
通读全部事件定义，**没有任何结构化表单回传或卡片交互事件** —— V4.0.0 spec 里设想的「H5 页面填写内容回传」在开放平台层面**没有对应的回调**，只能靠自由文本 + 上下文推断，或另做 H5 页面走自己的接口。

## 2. 三个前提，本项目当前两个不满足

### 2.1 ❌ 回调方向是「入站」，与现有集成相反

V4.0.0 确认的「本系统可与蓝信服务器接通」是**出站**（我们调它的 `apptoken/create`、`messages/create`）。
回调是**入站** —— 蓝信主动 POST 到我们，且有 **3 秒超时**。本系统是内网单机部署（nginx:80 → 127.0.0.1:8080），蓝信进不来。

### 2.2 ❌ 加解密要 AES，打破「纯 Python 标准库」

回调请求体是加密的：

```json
{ "dataEncrypt": "XXXXXXXX" }
```

算法为 **AES-CBC + PKCS 填充 + Base64**，URL 上另带 `timestamp` / `nonce` / `signature`（SHA1）。
蓝信自己的 Python 示例第一行就是 `pip install pycryptodome`。

而 **Python 标准库没有 AES**。本项目 V1.16.2 特意清除了所有第三方依赖，且要 PyInstaller 打包成单 exe。要走这条路只有两条选择：

- 引入 `pycryptodome`（破坏架构约束，打包体积与依赖管理都要改）
- 纯 Python 自实现 AES-CBC（约 200~300 行，回调量小性能可接受，但密码学自实现有风险）

### 2.3 ✅ 事件本身是够用的

`account_message` 能拿到「谁、什么时候、回复了什么」，足以支撑「回复归入跟进记录」或「已读回执」这类需求。

## 3. 内网环境下唯一可行的路径，以及它的代价

若入站始终不通，仍有一条**出站**路径：**订阅事件查询接口**

```
POST /v1/callback/events/fetch?app_token=APP_ACCESS_TOKEN
body: { "eventType": "account_message", "pageSize": 100, "eventOrgId": 524288 }
```

由我们主动去拉「推送失败后被持久化的事件」，返回**明文**（不需要 AES）。

**但代价必须写清楚**：

| 代价 | 说明 |
|---|---|
| **时延 ≥ 6 小时** | 蓝信要先重试 3 次（第 1 次失败后 **5 分钟 / 1 小时 / 6 小时**）全部失败，事件才会被持久化。用户回复后最快 6 小时才可能被我们拉到 |
| **事件只能查一次** | 文档原文：「一个事件只允许查询一次，通过查询接口完成查询后，事件会标记删除，再次查询时将不再返回」。**拉到之后我们处理失败，事件就永久丢了**，没有第二次机会 |
| 保留期未知 | 文档未说明失败事件持久化保留多久 |

对「用户回复后想尽快看到」这类诉求，6 小时时延基本等于不可用。
但对**组织架构同步**（`dept_*` / `staff_*`）而言，6 小时延迟是可接受的 —— 见 §4。

## 4. 将来若要重启这件事，决策树

```
蓝信凭证是否已申请下来并跑通推送?
├─ 否 → 先做这个。回调依赖同一套 appId/appSecret/网关地址
└─ 是 → 蓝信能否主动访问到本系统?
        ├─ 能(内网互通/反代暴露)
        │   → 走标准回调,实时可靠
        │   → 必须先解决 AES:引入 pycryptodome 还是纯 Python 自实现?
        │      (这一条要单独找用户确认,它动的是架构底线)
        └─ 不能
            → 只剩主动拉取,时延 ≥6h、事件只能查一次
            → 用于「回复归档」体验不可接受
            → 用于「组织架构同步」则可行:
               staff_create/modify/delete + dept_* 六个事件,
               6 小时延迟对组织架构变更完全够用,
               而且能顺带解决 V4.0.0 记的那个隐患 ——
               「蓝信人员编号是否等于我方工号」可由 staff 事件的
               staffId 与花名册工号建立映射,不必再人工加列
```

**注意**：`组织架构.xlsx` 目前是人工维护的，且 V4.0.0 已记录风险「若蓝信人员编号 ≠ 我方工号，需加一列人员编号」。若将来做通讯录同步，这条隐患可一并解决 —— 但那是独立的一件事，值得单独立项，不要和「回复回写」混在一起。

## 5. 文档站抓法（与 V4.0.0 那套不同，记下来省时间）

V4.0.0 调研用的 `developer.lanxin.cn` 是自研 SPA，正文要 `POST /api/v1/h5/document/fetch/detail` 带 `enName`。

**本次的 `openapi.lanxin.cn` 是 docsify**（纯前端 markdown 渲染），完全不同：

- 首页 `https://openapi.lanxin.cn/` 是 969B 的壳，`WebFetch` 只能拿到标题，抓不到正文
- 配置在 `https://openapi.lanxin.cn/docsify-config.js`，关键是 **`basePath: 'build-in'`**
- 因此 URL `#/server-api/callback/pushcallback/callback_api` 的正文在：
  ```
  https://openapi.lanxin.cn/build-in/server-api/callback/pushcallback/callback_api.md
  ```
  直接 `curl` 即可，**不需要任何逆向、不需要 Referer**
- 模块目录：`https://openapi.lanxin.cn/build-in/server-api/_sidebar.md`

本次取到的文档（均在 `build-in/server-api/callback/pushcallback/` 下）：
`outline.md` / `callback_events.md` / `callback_api.md` / `callback_api_encrypt_decrypt.md` / `events_fetch_api.md`；
另有 `callback/fetchcallback/`（拉取回调，用于工作台红点角标，**也是入站**，不是我们主动拉）与 `connector/`（连接器）。

## 6. 本轮结论

| 项 | 结论 |
|---|---|
| 蓝信是否提供回调 | **是** |
| 是否有「用户填写内容回传」的事件 | **有** `account_message`，但形态是自由文本回复，非结构化表单 |
| 本轮是否实现回调 | **否** —— 入站不可达 + 当前只需通知类 |
| 本轮是否改代码 | **是**，但与回调无关 —— 借这次复核发现推送侧一处 `fields` 单位歧义并修复，计 **V4.0.3**，详见 §7 |
| 前置卡点 | ① 蓝信凭证仍未申请（推送功能自 V4.0.0 起至今未联调）② 入站可达性 ③ AES 依赖与纯标准库架构的冲突 |

## 7. 附：借这次复核，对推送代码做的一次全量核对（2026-07-20）

用 `openapi.lanxin.cn` 的文档重新核对了 V4.0.0 基于 `developer.lanxin.cn` 写的推送实现。

### 7.1 完全一致、无需调整的部分

| 项 | 新文档 | 当前代码 |
|---|---|---|
| 取 token | `GET /v1/apptoken/create?grant_type=client_credential&appid=&secret=` | 一致 |
| `expiresIn` | 7200 秒，建议缓存 | 缓存 + 300s 提前量 |
| 工号换 ID | `GET /v2/staffs/id_mapping/fetch?app_token=&org_id=&id_type=&id_value=` | 一致 |
| `id_type` 枚举 | employ_id / mobile / mail / login / external_id | 一致 |
| 发消息 | `POST /v1/messages/create?app_token=` | 一致 |
| `msgType` | text / oacard / linkCard / appCard | 一致 |
| `userIdList` 上限 | 1000 | `MAX_RECIPIENTS = 1000` |
| `invalidStaff` 返回 | 无效人员ID列表 | **已处理**，转 `LanxinError` 计入 failed |
| `msgId` 返回 | 消息ID | 已收集进 `msgIds` |
| `bodyTitle` | 200×3 = 600 字节 | 600 |
| `bodySubTitle` | 400×3 = 1200 字节 | 1200 |
| `bodyContent` | 1000×3 = 3000 字节，八行 | 3000 |
| `fields` 对数 | ≤10 | `MAX_FIELDS = 10` |
| `10005` | API服务 无权限 | 不重试（正确） |
| `56008` | 触发限流 | 退避重试 1/2/4s（正确） |

错误码全表 115 条，限流类只有 `56008`（`40053` 是分片、`57000` 是短信），代码判断无遗漏。

### 7.2 唯一需要调整的：`fields` 的单位歧义（已修，V4.0.3）

两版文档对 `fields` 的表述不同：

| | `fields.key` | `fields.value` |
|---|---|---|
| `developer.lanxin.cn`（V4.0.0 依据） | 18 **字节** | 192 **字节** |
| `openapi.lanxin.cn`（本次） | 不超过 6 个**汉字** | 两行（接口限制 64 **字**） |

纯中文时等价（6×3=18、64×3=192），**混合中英文时不等价**。V4.0.0 为绕开 18 字节定的两个短标签正好踩中：

```
成本超支>5k   7 字符 / 15 字节   ← 按字符超限、按字节合规
成本超支<5k   7 字符 / 15 字节
```

**已修**：新增 `fit_field(s, max_chars, max_bytes)`，字符数与字节数同时满足、取更严的一个；两个短标签改为 `超支>5千` / `超支<5千`（5 字符 / 11 字节）。`build_summary_card` 里 value 的丢弃循环也改成双重判断——`value` 是「N 项：标签 n · 标签 n」的混合文本，64 字符比 192 字节严格得多。

修完全部 16 个 `fields.key`（8 类关注原因 + 8 类工时问题）都 **≤6 字符且 ≤18 字节**，两种解读下都不越限。

`bodyTitle` / `bodySubTitle` / `bodyContent` 保持用 `fit_bytes`——那三个字段两版文档都明写字节，不需要双重限制。

### 7.3 新文档里有、当前未使用的能力（备查）

- **`cardLink` / `pcCardLink`**：卡片整体的点击跳转地址。最初「推送消息给指定人员（H5 页面）」的设想，这个字段就能实现，前提是内网有蓝信客户端能打开的地址
- **`departmentIdList`（≤100）**：可直接按分支发送，整个 L4 组一次发完，不必逐人换 staffId。全组织广播填根分支 `orgId-0`
- **撤回消息** `/server-api/message/revoke_message.md`：V4.0.0 记为「事故预案未实现」，接口是存在的
- **动态卡片**：`isDynamic` + `headStatusInfo` + 更新状态接口，适合审批类「待处理 → 已处理」
- **富文本**：`bodyTitle` 等支持 `<div style="color:#5A83E9;font-size:18pt;text-align:left">`，可控颜色/字号/对齐/首行缩进
- **`links`（≤3 对）**、**`staffId`**（卡片展示发起者头像）
- 另有 `create_notification_message`（应用通知消息）、`create_staff_private`（人员私聊提醒）两个别的通道
