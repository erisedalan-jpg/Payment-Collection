# 蓝信接口与回调地址 —— 填写说明（2026-07-20）

给**申请蓝信应用、以及凭证到位后做配置**的人用。分两部分：

- **§1 你要去蓝信开发者中心填/取什么**
- **§2 你要在本系统 `/data` 页填什么**

> **一句话结论（回调部分）**：本系统**不订阅任何回调事件**，所以蓝信开发者中心里的
> 「回调地址」「动态数据拉取地址」两个框**都留空**。原因见 §3。

---

## 1. 蓝信开发者中心（登录 `open.lanxin.cn` 或贵司的私有部署地址）

### 1.1 要「取」的四项 —— 填进本系统

在**应用详情页**取得，缺一不可：

| 项 | 在哪取 | 形态示例 |
|---|---|---|
| **AppId** | 应用详情页 | `2990080-14155776` |
| **AppSecret** | 应用详情页 | 一串随机字符 |
| **组织ID（orgId）** | 应用详情页；**若详情页不给**，可由任意 staffId 的横杠前段反推 | `524288` |
| **网关地址（apiGateway）** | 应用详情页 / 问组织管理员 | `https://apigw.你们公司域名` |

> **网关地址一定要问清楚。** 蓝信文档里所有示例写的是 `https://apigw-example.domain`，
> **那是占位符**，真实地址随部署实例不同。填错的表现是自检第 ① 步就失败。

### 1.2 要「填」的：回调地址 —— **本系统留空**

应用详情页 →「事件回调」，有两个框：

| 框 | 本系统怎么填 |
|---|---|
| 回调地址（订阅事件推送） | **留空** |
| 动态数据拉取地址 | **留空** |

不填的后果：**没有任何后果**。不订阅事件就不会有推送，这两个地址只在订阅事件时才被调用。

**若将来要填**，格式是标准 HTTPS URL，本系统的话会是：

```
https://<蓝信服务器能访问到的本系统地址>/api/lanxin/callback
```

注意三点（这也是本轮决定不做的原因，详见 `docs/2026-07-20-蓝信回调接口调研.md`）：

1. 该地址必须**蓝信服务器能主动访问到**，且响应要在 **3 秒内**
2. 请求体是 **AES-CBC 加密**的，需要解密 + 验 SHA1 签名
3. 本系统当前是内网部署，蓝信进不来

### 1.3 要「开通」的三件事 —— 都得找组织管理员

开发者本人做不了，必须由组织管理员操作：

| 事项 | 不做的后果 |
|---|---|
| 开通「**组织开发者**」身份 | 连应用都创建不了（**这是最前面的卡点**） |
| 开通 **`messages/create`** 接口权限 | 自检第 ③ 步失败，报错码 `10005`「API服务 无权限」 |
| 关闭「**测试模式**」并设置可见范围 | 自建应用创建即发布，但默认在测试模式，正式推送前要关掉 |

## 2. 本系统 `/data` →「配置」签 →「蓝信推送」卡（仅超管可见）

把 §1.1 取到的四项填进去，点「保存配置」。四个框与蓝信侧的对应关系：

| 本系统的框 | 填 §1.1 的哪一项 |
|---|---|
| AppId | AppId |
| AppSecret | AppSecret（保存后不回显，只显示「已配置」） |
| 组织ID | orgId |
| 网关地址 | apiGateway，**必须以 `https://` 开头**，结尾的 `/` 会被自动去掉 |

保存后在「自检工号」填**你自己的工号**（如 `A000701`），点「**连通性自检**」。三步全绿即接入完成：

| 步骤 | 实际调的接口 | 失败意味着什么 |
|---|---|---|
| ① 取应用访问TOKEN | `GET {网关}/v1/apptoken/create?grant_type=client_credential&appid=&secret=` | AppId / AppSecret / 网关地址有误 |
| ② 工号换人员ID | `GET {网关}/v2/staffs/id_mapping/fetch?app_token=&org_id=&id_type=employ_id&id_value={工号}` | 组织ID 有误，**或蓝信「人员编号」≠ 我方工号**（见下） |
| ③ 发测试消息给本人 | `POST {网关}/v1/messages/create?app_token=` | `messages/create` 权限未开通（错误码 `10005`） |

自检全程**只给你自己发消息，不触碰任何他人**。

> **⚠ 第 ② 步失败且组织ID确认无误时**：说明蓝信通讯录的「人员编号」与我方 `组织架构.xlsx`
> 的「工号」不是同一套编码。
> 应对：请贵司提供全员「工号 ↔ 人员编号」对照，我方在 `组织架构.xlsx` 增加「人员编号」列并发新版本解决。
> **改用手机号/邮箱匹配不可行** —— 现有 `组织架构.xlsx` 不含这两列。
> （蓝信 `id_type` 另支持 `mobile`，但格式要求带国家码，如 `86-13723129089`。）

## 3. 为什么回调地址留空

完整调研见 `docs/2026-07-20-蓝信回调接口调研.md`，摘要：

- 蓝信**确实提供**回调，共 14 种事件，其中 `account_message`（人员回复应用号消息）能拿到用户的文字回复
- 但回调是**入站**（蓝信主动 POST 到我们，3 秒超时），而本系统内网部署、蓝信访问不到
- 且请求体需 **AES-CBC 解密**，而本项目是纯 Python 标准库、无 AES
- 当前诉求是**通知类即可、暂不做回写**

所以两个回调地址框留空即可。将来若要启用，是一次独立立项，前提是先把「入站可达性」和「AES 依赖」两件事定下来。
