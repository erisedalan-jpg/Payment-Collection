# 蓝信接口与回调地址 —— 填写说明（2026-07-20，V4.0.5 改写）

给**申请蓝信应用、以及凭证到位后做配置**的人用。分三部分：

- **§1 你要去蓝信开发者中心填/取什么**
- **§2 你要在本系统 `/data` 页填什么**
- **§3 员工回复去哪了**

> **一句话结论**：本系统**要订阅回调事件**，「订阅事件回调地址」填
> `https://<部署地址>/api/lanxin/callback`（`/data` 页有现成的复制按钮），
> 「动态数据拉取地址」留空。
>
> 此前版本的本文写「两个框都留空」，那是 V4.0.5 之前的结论，**已作废**。

---

## 1. 蓝信开发者中心（登录 `open.lanxin.cn` 或贵司的私有部署地址）

### 1.1 要「取」的四项 —— 填进本系统

在**应用详情页**取得，缺一不可：

| 项 | 在哪取 | 形态示例 |
|---|---|---|
| **AppId** | 应用详情页 | `2990080-14155776` |
| **AppSecret** | 应用详情页 | 一串随机字符 |
| **组织ID（orgId）** | 应用详情页；**若详情页不给**，可由任意 staffId 的横杠前段反推 | `524288` |
| **网关地址（apiGateway）** | 应用详情页 / 问组织管理员 | `https://apigw.你们公司域名` |

> **网关地址一定要问清楚。** 蓝信文档里所有示例写的是 `https://apigw-example.domain`，
> **那是占位符**，真实地址随部署实例不同。填错的表现是自检第 ① 步就失败。

### 1.2 要「填」的：回调地址（V4.0.5 起**要填了**）

应用详情页 →「事件回调」，有两个框：

| 框 | 本系统怎么填 |
|---|---|
| 订阅事件回调地址 | `https://<部署地址>/api/lanxin/callback` |
| 动态数据拉取地址 | **留空**（本系统不使用） |

`<部署地址>` 就是你平时访问本系统的地址。**不确定填什么时**：用超管账号打开 `/data` →
「配置」签 → 蓝信推送卡，那里有一个「回调地址」框会**自动拼好**，直接点复制即可。

前提：该地址必须**蓝信服务器能主动访问到**（蓝信是私有化部署在贵司内网，正常可达），
且本系统需在 **3 秒内**响应——已实现，实测为毫秒级。

### 1.3 要「订阅」的事件

同一页面往下，事件列表里点「编辑」开启订阅并保存。**按发送身份选**：

| 发送身份 | 要订阅的事件 |
|---|---|
| 应用号（默认） | `account_message`（人员给应用公众号发消息） |
| 智能机器人 | `bot_private_message` + `bot_group_message` |

**不订阅的后果**：推送照常发出，但员工的回复永远收不到，收件箱恒为空。

### 1.4 要「取」的另外两项凭证

**同样在「事件回调」页**（不是应用详情页，与 AppId/AppSecret 是**不同的凭证**，别混）：

| 项 | 用途 |
|---|---|
| **回调密钥（aesKey）** | 解密回调报文体。43 位字符串 |
| **回调签名令牌（signToken）** | 校验请求确实来自蓝信 |

**签名令牌填错的表现**：回调全部被拒、收件箱一直空。此时打开 `/data` 的蓝信卡，
会显示「**已拒绝 N 次验签失败**」——这是判断令牌填错的唯一线索，看到它就说明令牌不对。

### 1.5 要「开通」的三件事 —— 都得找组织管理员

开发者本人做不了，必须由组织管理员操作：

| 事项 | 不做的后果 |
|---|---|
| 开通「**组织开发者**」身份 | 连应用都创建不了（**这是最前面的卡点**） |
| 开通 **`messages/create`** 接口权限 | 自检第 ③ 步失败，报错码 `10005`「API服务 无权限」 |
| 关闭「**测试模式**」并设置可见范围 | 自建应用创建即发布，但默认在测试模式，正式推送前要关掉 |

## 2. 本系统 `/data` →「配置」签 →「蓝信推送」卡（仅超管可见）

把 §1.1 的四项和 §1.4 的两项填进去，点「保存配置」：

| 本系统的框 | 填哪一项 |
|---|---|
| AppId | §1.1 的 AppId |
| AppSecret | §1.1 的 AppSecret（保存后不回显，只显示「已配置」） |
| 组织ID | §1.1 的 orgId |
| 网关地址 | §1.1 的 apiGateway，**必须以 `https://` 开头**，结尾的 `/` 会被自动去掉 |
| 回调密钥 | §1.4 的 aesKey（同样不回显） |
| 回调签名令牌 | §1.4 的 signToken（同样不回显） |
| 发送身份 | 默认「应用号」。**除非组织管理员已开通机器人能力，否则不要改成「智能机器人」** |

> **三个密钥框留空 = 不修改**（不是清空）。因为读取时它们一律脱敏成空串，
> 若「留空即清空」，任何一次「只改网关地址」的保存都会把已配好的密钥抹掉。
> 要更换密钥就填新值；要保持不变就留空。

**「回调地址」框是只读的**，系统按你当前访问的地址自动拼好，点旁边的复制按钮
即可粘到 §1.2 的蓝信侧。

保存后在「自检工号」填**你自己的工号**（如 `A000701`），点「**连通性自检**」。三步全绿即接入完成：

| 步骤 | 实际调的接口 | 失败意味着什么 |
|---|---|---|
| ① 取应用访问TOKEN | `GET {网关}/v1/apptoken/create?grant_type=client_credential&appid=&secret=` | AppId / AppSecret / 网关地址有误 |
| ② 工号换人员ID | `GET {网关}/v2/staffs/id_mapping/fetch?app_token=&org_id=&id_type=employ_id&id_value={工号}` | 组织ID 有误，**或蓝信「人员编号」≠ 我方工号**（见下） |
| ③ 发测试消息给本人 | `POST {网关}/v1/messages/create?app_token=` | `messages/create` 权限未开通（错误码 `10005`） |

自检全程**只给你自己发消息，不触碰任何他人**。

> **⚠ 第 ② 步失败且组织ID确认无误时**：说明蓝信通讯录的「人员编号」与我方 `组织架构.xlsx`
> 的「工号」不是同一套编码。
> 应对：请贵司提供全员「工号 ↔ 人员编号」对照，我方在 `组织架构.xlsx` 增加「人员编号」列并发新版本解决。
> **改用手机号/邮箱匹配不可行** —— 现有 `组织架构.xlsx` 不含这两列。
> （蓝信 `id_type` 另支持 `mobile`，但格式要求带国家码，如 `86-13723129089`。）

## 3. 员工回复去哪了

配好回调后，员工在蓝信里直接回复推送卡片，内容会进入
`/data` →「**蓝信回复**」签（仅超管可见）。

- 每条显示：接收时间 / 姓名 / 工号 / 来源（私聊·群聊·应用号）/ 回复正文
- **身份查不到时显示「未知」并列出原始 staffId**，不会用编号冒充姓名
- **读不懂的报文不会被丢弃**，而是标成「未解析」并写明原因——这是我们排查真实报文
  形态的唯一线索，请不要忽略它们
- 超管可把一条回复「**归入**」到风险跟进 / 临时重点跟进 / 回款重点跟进 / 重点项目进展，
  归入是**追加**到既有跟进内容之后，不会覆盖别人已写的东西

> **归入时的候选项目是「推测」不是结论。** 蓝信回调不携带任何原卡标识，
> 系统只能按「最近推给这个人的卡片涉及哪些项目」来猜。下拉里可以选任意项目。

**倚天工时问题的回复只留在收件箱、不提供归入**——工时问题是人维度的，
现阶段没有对应的跟进表。

全绿后**建议这样开第一枪**：

1. 先把两条路由的汇总级别都设为 0、**只发给自己**，看清卡片实际观感
2. 自己在蓝信里回一条，确认它出现在「蓝信回复」签里
3. 再逐步放开到上级

收件人是真人，第一条发出去的观感很难重来。而且**这套代码至今没有跑过一条真实数据**——
第一次联调请预留排查时间，特别留意收件箱里的「未解析」条目：它们会告诉你
真实报文与文档的差异在哪。
