# V2.6.10 成本预算口径兜底 + 数据质量告警 设计

> 版本：V2.6.10（Z 级，子页面口径调整）
> 日期：2026-07-05
> 背景来源：用户在 `/project/:id` 预算核算页发现项目 `QAGD-SS-202603249009` 的 PMIS「总预算」= 5640（等于概算），低于损益表「预算成本」7728.5，质疑总预算取值是否异常。

## 1. 问题定性（已核实）

对全仓 638 个项目全量比对（脚本核对真实 `data/analysis_data.json` + 原始 `input/pmis/项目状态信息数据.xlsx`）：

- **平台读取无 bug**：`cost.总预算` 忠实读取 PMIS《项目状态信息数据》「项目总预算（元）」列原值（`QAGD-SS-202603249009` 原始 xlsx 值即 5640.0）。这是设计钦定口径（`pmis.py` 注释：「成本分析用项目状态信息总预算」）。
- **但存在源数据口径分歧**：PMIS 在两个子系统里对同一项目给了两个不同的预算数——《项目状态信息》的「项目总预算」与损益表 `profit_loss_direct.csv` 的「预算成本」。
  - 正常一致：**631 / 638（98.9%）**。
  - **分歧：7 个项目**，其中 6 个 PMIS「项目总预算」恰好塌到「概算」值（更低），抬高了消耗比；个别项目超支状态因此翻转。

## 2. 目标与边界

- **只解决"分歧"**：当项目**同时有** PMIS总预算 与 损益预算成本、且 `|PMIS总预算 − 损益预算成本| > 1 元` 时，把成本口径分母改用**损益预算成本**并连带重算。
- **一致的保留原样**：一致的 631 个（含浮点噪声，差 ≤ 1 元）、以及只有单边数据的项目，一律保留 PMIS 原值，**结果字节不变**。
- **不扩范围（YAGNI）**：PMIS总预算为空的项目**不**用损益值去填（避免改动约 342 个空成本项目）。
- **只动分母**：分子「核算」始终保持 PMIS核算（已核对这 7 个项目 PMIS核算 与损益实际成本一致，无歧义）。
- **判据统一**：同一分歧判据（`|差| > 1 元` 且两值皆存在）同时驱动"口径覆盖"与"数据质量告警"。

## 3. 实现设计

### 3.1 落点
`derive_cost`(`pmis.py`) 只有 PMIS 表、拿不到损益数据，**不动它**。在 `preprocess_data.py` 组装 `final_data`（约 line 240）**之前**新增一个纯函数：

```
reconcile_cost_budget(project_pmis: dict, project_profit: dict | None) -> list[dict]
```

- 遍历 `project_pmis[pid]["cost"]`，取 `project_profit[pid]["summary"]["预算成本"]` 作为 `profit_budget`。
- 判据：`pmis_total is not None and profit_budget is not None and abs(pmis_total - profit_budget) > 1.0`。
- 满足则**就地覆盖并连带重算**该项目 cost dict：
  - `总预算 = profit_budget`
  - `剩余预算 = 总预算 − 核算`（不再沿用 PMIS 那列；核算为 None 时剩余为 None）
  - `消耗比 = 核算 / 总预算`（`总预算 > 0 and 核算 is not None` 否则 None）
  - `项目超支 = (剩余预算 is not None and 剩余预算 < 0)`
  - `成本状态` = **PMIS 原文不动**
  - `交付超支` = PMIS flag 不动
- 返回分歧清单 `[{"projectId", "pmisBudget", "profitBudget", "diff"}]`（供告警使用）。
- `project_profit` 为 None（无损益数据）时函数直接返回 `[]`、不改任何 cost。

### 3.2 数据质量告警
照批2 `dataQuality.collectionParseErrors` 同构，在 `dataQuality` 增字段 `budgetSourceMismatch`：

```
"budgetSourceMismatch": {
  "count": <int>,
  "items": [{"projectId", "pmisBudget", "profitBudget", "diff"}, ...]
}
```

- 由 `reconcile_cost_budget` 的返回清单填充。
- schema 用**宽松扩展**（pydantic `extra="allow"`，同批2），不算结构破坏，无需前端 `gen:types`。
- 前端治理/数据质量页展示：汇总计数 + 项目清单（pid / PMIS总预算 / 损益预算成本 / 差额）。沿用该页既有告警的展示范式。

### 3.3 前端
- `cost.总预算/剩余预算/消耗比/项目超支` 为后端已算值，`/project/:id` 预算核算页无需改逻辑，重算后数值自然生效。
- 治理页新增分歧清单的渲染（读 `dataQuality.budgetSourceMismatch`）。

## 4. 影响面（实测）

- **仅 7 个项目数值变化，631 个字节不变。**
- **超支项目数 36 → 35**：`WSGF-SS-202603259038` 由「超支(106.8%)」翻为「不超支(99.4%)」；其余 6 个不改变超支侧，仅消耗比下降几个点。
- 消费方主要是 `/project/:id` 预算核算页的 `costSummary`。首页 / `/projects`「成本超支」卡走 `overspendAmount`（损益域，`实际成本 − 预算成本`），**不受本改动波及**——改完两套口径反而更靠拢（PMIS 汇总块的分母也变成损益预算成本）。

## 5. 测试与验证

**先补测试再改实现（TDD，改 preprocess 计算强制要求）：**

`tests/` 新增对 `reconcile_cost_budget` 的单测，覆盖：
1. 分歧项目：覆盖 `总预算`、重算 `剩余预算/消耗比/项目超支`，`成本状态/交付超支` 不动。
2. 一致项目（差 ≤ 1 元）：cost 完全不变。
3. 只有 PMIS总预算、无损益预算成本：不变。
4. 只有损益预算成本、PMIS总预算为空：不变（不填充）。
5. 核算为 None 的安全性：剩余/消耗比为 None、不抛异常。
6. 告警清单：返回的 items 字段正确、count 一致。
7. 用真实 7 项目做契约冒烟（如 `QAGD-SS-202603249009` 改后消耗比 ≈ 65.2%、剩余 ≈ 2691.5、不超支；`WSGF-SS-202603259038` 改后不超支）。

**验证：**
- `bash verify.sh` 全绿（语法 + ruff + pytest + 前端 typecheck/vitest/build）。
- 真机启动（`python server.py` + 点「更新数据」）核对：`QAGD-SS-202603249009` 消耗比 65.2%、`WSGF-SS-202603259038` 不超支、治理页出现分歧清单（7 项）。
