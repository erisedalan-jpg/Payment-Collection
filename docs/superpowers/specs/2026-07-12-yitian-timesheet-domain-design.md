# 倚天工时域（/yitian）设计 — V3.0.0

> 状态：设计已确认，待落 plan
> 目标版本：**V3.0.0**（X 级大版本，用户已确认）
> 前置基线：V2.10.0（线上）

## 1. 背景与目标

`yitian-analyze/` 下有一个独立工具（`timesheet_checker_v3.py`，4437 行）：读倚天导出的 `工时.xlsx`，做工时填写合规性检查与统计分析，拼出一个单体 HTML 报告。它有三个结构性缺陷：

1. **写死 2026**：日历（工作日/计算周 W1..W52）以 366 行常量内嵌在脚本里；「服务方式」检查还硬编码了一句「2026-05-09 之前不检查」。跨年即失效。
2. **组织/人员取自工时表自身**：用工时表的 `L3组织/L4组织` 列分组，用 `员工`（姓名）做 join。本仓已有权威花名册 `input/组织架构.xlsx`，且实测**按姓名 join 只中 84/85，按工号 join 中 85/85**——姓名 join 会静默丢人。
3. **产物是死 HTML**：算完即渲染，无法按任意周期重看，无权限隔离，与本平台数据/权限体系完全割裂。

**本期目标**：不是移植，是**重构**——只保留其分析口径（合规检查、工时统计、趋势、TOP1000 大客户支持、跨 BG 支持），在本平台内以「离线导入 → 后端管线 → 独立 JSON → Vue3 子域页面」的形态重建，年度无关、权限受控、周期动态可选。

**非目标**（本期不做）：
- 倚天 cookie 自动下载脚本（尚未完善）。本期只做离线导入，但把落盘目录 `input/yitian/` 与「更新数据」流程的接缝留好，脚本完成后往该目录落文件即可无缝接上。
- 合规规则库的可视化维护界面（规则以 py 常量维护，改一处即可）。
- 删除 `yitian-analyze/` 目录（本期保留作口径比对基准，上线核对后再清）。

| 来源 | 位置 | 角色 | 冲突时 |
|---|---|---|---|
| 倚天工时导出 | `input/yitian/工时.xlsx`（**新增**，页面上传） | 工时事实行（77 列，取白名单 14 列） | — |
| 组织架构 | `input/组织架构.xlsx`（已有） | **人员与组织的唯一权威**：85 人花名册（工号/姓名/新L2/新L3/新L3-1/新L4/员工类别） | **以花名册为准**，工时表自带的 `L2/L3/L3-1/L4组织` 列忽略 |
| TOP1000 客户 | `input/TOP1000.xlsx`（已有） | TOP1000 大客户判定 | **以系统为准**，`yitian-analyze/TOP1000客户清单.xlsx` 弃用 |
| 节假日表 | `input/yitian/holidays.csv`（**新增**，页面上传） | 工作日口径（跨年） | 缺失则降级（见 §5） |

**连接键 = 工号**（工时表 `员工编号` 小写 `a000701`，花名册 `工号` 大写 `A031492` → 统一 `.strip().upper()` 归一后 join）。姓名仅用于展示，**不作为连接键**（花名册存在「马敏02」这类重名消歧后缀；实测姓名 join 丢 1 人）。

**工时表取列白名单**（其余 63 列一律不读、不落盘）：

`ID` / `工时ID`（记录标识）、`员工编号`、`员工`、`工时类型`、`工时`、`工作日`、`工作成果`、`客户`、`项目类型`、`工作类型三`、`产研侧产品线`、`产研侧产品名称`、`工单编号`、`销售L2组织`、`服务方式`。

> **隐私护栏**：`员工电话` / `员工所在省` / `员工所在市` / `员工入职省份` / `员工入职城市` / `岗位` 等个人字段**禁止读取、禁止落 JSON**。`工作成果`（均长 196 字）**只对问题行下发前 120 字摘要**，合规行不下发正文。

**花名册 = 应填人员全集**：`roster − 有工时记录的工号` = **一条都没填的人**。原工具看不到这类人（它只能看到"填了但不足"），这是重构后白捡的增益。

## 3. 架构与数据流

```
input/yitian/工时.xlsx     ┐
input/yitian/holidays.csv  │
input/组织架构.xlsx        ├─ yitian.py (解析 + 工作日 + 合规判定 + 精简化)
input/TOP1000.xlsx         ┘        │  ← 复用 projects.read_org_names / read_top1000
                                    │  ← 规则常量来自 yitian_rules.py
                                    v
                          data/yitian_data.json   (独立产物，不进 analysis_data.json)
                                    │  GET /api/yitian/data  (按 allowedL4 服务端切数据)
                                    v
   /yitian 总览 + /yitian/{compliance,analytics,trend,customer}
   前端 stores/yitian.ts 惰性加载 + lib/yitian/*.ts 纯函数按任意区间现算
```

**新增后端模块**
- `yitian_rules.py` — 纯数据常量：必填字段同义词表、工时类型禁止词表、产品线关键词表（22 条）、二级产研产品名关键词表、排除词、本 BG 销售 L2 组织清单、服务方式生效日。规则改动只碰这一个文件。
- `yitian.py` — 纯函数管线：读源 → 工号 join 花名册 → 工作日/双周标签 → 数据校正 → 合规判定 → 码表压缩 → 输出 `YitianData`。

**接入现有管线**：`preprocess_data.py` 末段追加倚天段——`input/yitian/工时.xlsx` 存在才跑，产出 `data/yitian_data.json`；**文件不存在则跳过并打 `[INFO]`，绝不阻断主管线**（主域数据与倚天域完全解耦）。用户仍然只有一个「更新数据」按钮。

**契约**：`schema.py` 新增 `YitianData` pydantic 模型（与 `AnalysisData` 并列的第二个根模型），导出 JSON Schema → `cd frontend && npm run gen:types` 生成 `frontend/src/types/yitian.ts`。

## 4. 数据契约（`data/yitian_data.json`）

码表（string interning）+ 精简事实行。**前端拿到 `days + entries` 即可按任意日期区间/组织/类型重算全部指标**——动态周期是结构自带的，不是加出来的。

```jsonc
{
  "meta": {
    "periodStart": "2026-04-17",      // 工时数据最早日期(= days[] 覆盖范围下界)
    "periodEnd":   "2026-04-23",      // 最晚日期(= days[] 覆盖范围上界)
    "generatedAt": "2026-07-12T10:00:00",
    "rows": 540,                       // entries 行数(含业务类/假期类/管理类,不预过滤)
    "employees": 85,                   // 花名册人数(L4 切分后为可见人数)
    "calendarSource": "csv"           // "csv" | "fallback"(holidays.csv 缺失,降级纯周一~周五)
  },
  "roster": [                          // 花名册,应填人员全集
    { "id": "A031492", "name": "马敏02", "l2": "交付中心", "l3": "交付实施三部",
      "l31": "服务二部", "l4": "广东二服务组", "category": "正式员工" }
  ],
  "days": [                            // periodStart..periodEnd 逐日,双周标签预打
    { "d": "2026-04-17", "workday": true, "isoWeek": "2026-W16", "calcWeek": "2026-CW17" }
  ],
  "dims": {                            // 码表(entries 里按下标引用)
    "types":        ["项目类", "售前类", "售后类", "管理类", "业务类", "假期类"],
    "workTypes":    ["安装部署", "..."],   // 工作类型三
    "customers":    ["江苏省税务局", "..."],
    "products":     ["NGSOC", "..."],      // 产研侧产品线
    "productNames": ["奇安信网神SSL编排控制网关系统V6.0", "..."],  // 产研侧产品名称
    "projectTypes": ["售前服务", "..."],
    "salesL2":      ["小金融集团军", "..."],
    "serviceModes": ["客户现场", "远程", "电话"]
  },
  "entries": [                         // 精简事实行(不含工作成果正文)
    { "d": "2026-04-17",   // 工作日
      "e": "A031492",      // 工号(→ roster)
      "t": 0,              // 工时类型 → dims.types
      "h": 6.0,            // 工时
      "wt": 3,             // 工作类型三 → dims.workTypes (null 可)
      "cu": 12,            // 客户 → dims.customers (null 可)
      "pl": 5,             // 产研侧产品线 → dims.products (null 可)
      "pn": 1,             // 产研侧产品名称 → dims.productNames (null 可)
      "pt": 0,             // 项目类型 → dims.projectTypes (null 可)
      "sm": 0,             // 服务方式 → dims.serviceModes (null 可)
      "bg": 2,             // 销售L2组织 → dims.salesL2 (null 可)
      "wo": "WO2026041700123",  // 工单编号(同工单关联检查用)
      "top": true,         // 该行客户 ∈ TOP1000
      "chk": true,         // 是否进入合规检查(= 工时类型 ∉ {业务类,假期类} 且 工时 > 0)
      "ok": 2,             // 合规状态 0=合规 1=合规(提示) 2=问题 (chk=false 时恒 0,不计入合规率)
      "iss": ["MISS_SERVICE_MODE"]   // 问题码列表(空=无问题)
    }
  ],
  "issues": [                          // 仅问题行(ok≠0)的明细
    { "i": 17,                          // entries 下标
      "codes": ["MISS_SERVICE_MODE"],
      "msgs": ["缺少服务方式"],
      "snippet": "本周完成XX客户天擎V10升级……"   // 工作成果前 120 字
    }
  ]
}
```

**`entries` 不预过滤**：保留工时表的全部有效行（含业务类/假期类/管理类/0 工时），由 `chk` 标记是否进入合规检查。原因：工时统计（总工时、类型占比）需要业务类/假期类，而合规率不能含它们——两套口径共用一份事实行，靠 `chk` 区分，避免下发两份数据。

**`days` 覆盖范围** = `[meta.periodStart, meta.periodEnd]`（工时数据的实际日期跨度）。前端日期区间选择器**必须钳制在此范围内**——超出范围没有工作日标注，基础工时算不出来。

**体积估算**：全年约 2.8 万事实行 × ~55B ≈ 1.5MB，问题行摘要 ≈ 0.8MB，合计 **~2.3MB**，且**仅在进入 /yitian 时拉一次**（对比首页 `analysis_data.json` 现已 16MB，倚天域不额外拖慢全站）。

## 5. 工作日与双周口径

### 5.1 工作日（跨年，数据驱动）

`input/yitian/holidays.csv`，UTF-8，表头 `日期,类型`：

```csv
日期,类型
2026-01-01,休
2026-02-16,休
2026-02-14,班
```

- `休` = 法定假日或调休放假日（即使是周一~周五）
- `班` = 调休上班日（即使是周六/周日）
- **工作日判定** = （周一~周五 且 不在「休」）**或**（在「班」）
- 一年只需 ~25 行；跨年只需补行，**不改代码、不重新打包**
- **文件缺失/解析失败 → 降级**：`calendarSource: "fallback"`，工作日 = 纯周一~周五，`/yitian` 各页顶部展示告警条（提示节假日表缺失，饱和度在含节假日的周期会偏低）。降级是设计行为，不是错误——与本仓「空人员清单 = 不过滤，降级」同款风格。
- `holidays.csv` 覆盖范围之外的日期（如数据跨到没维护的年份）：按纯周一~周五处理，并在 meta 里标注（`calendarSource: "csv"` 但页面告警提示覆盖范围）。

### 5.2 双周口径（可切换）

后端在 `days[]` 上**同时预打两种周标签**，前端切换只是换 groupBy 键，零重算风险：

- `isoWeek`：ISO 自然周（周一~周日），标签 `2026-W16`
- `calcWeek`：倚天计算周（**上周五 ~ 本周四**，与倚天填报截止口径一致），标签 `2026-CW17`
  - 推导规则（年度无关）：某日所属计算周 = 该日**向后推至最近的周四**所在的 ISO 周序号 + 前缀 `CW`。即 周五/周六/周日/周一/周二/周三/周四 归入同一个 CW。不依赖任何写死的 W1..W52 表。

**切换器作用域**：全局（Pinia store），**按登录账号持久化**（复用 V2.8.3 的 `userScopedKey` localStorage 范式）。影响 `/yitian` 全部子页的周分组与趋势图 X 轴。

**日期区间**：主口径是**任意起止日期**（复用全站 FilterBar 日期区间惯例，V1.11.0）。周口径只用于「按周聚合展示」（趋势图 X 轴、周维度汇总表）。

## 6. 合规规则库（`yitian_rules.py`）

规则逐字迁自 `yitian-analyze/timesheet_checker_v3.py`（`CHECK_RULES` L23-47、`TYPE_MISMATCH_RULES` L52-80、`PRODUCT_LINE_KEYWORDS` L335、`PRODUCT_NAME_KEYWORDS` L364），**唯一例外是「服务方式」改判（§6.3）**。

### 6.0 数据校正（在过滤与检查之前）

`项目类型` 含「售前服务」→ **`工时类型` 强制改为「项目类」**（确保售前服务项目进项目类检查与统计口径）。校正同时作用于检查数据与统计数据。

### 6.1 检查范围

| 工时类型 | 处理 |
|---|---|
| 项目类 / 售前类 / 售后类 | 全量检查 |
| 管理类 | 跳过检查，直接标记合规（仍进工时统计） |
| 业务类 / 假期类 | 过滤阶段剔除（不进检查、不进合规率；**仍进总工时/类型占比统计**） |
| 工时 = 0 | 跳过检查 |

### 6.2 必填字段（全文模糊匹配 `工作成果`）

适用：项目类 / 售前类 / 售后类。命中任一同义词即视为已填。

| 字段 | 问题码 | 同义词 |
|---|---|---|
| 工作概述 | `MISS_SUMMARY` | 工作概述、工作概况、工作总结、工作汇报、工作总述、工作述职 |
| 工作进展 | `MISS_PROGRESS` | 工作进展、工作进度、工作内容、进展情况、进行的工作、今日工作、工资进展、工资进度、工作已进、已完成工作、工作完成情况、工作当前 |
| 下一步工作计划 | `MISS_NEXT` | 下一步工作计划、下一步计划、下一步工作、下一步、后续计划、明日计划、工作计划、下步计划、下步工作、后续工作、之后计划、下期计划 |

### 6.3 服务方式（**本次口径修正**）

**原实现是缺陷**：去 `工作成果` 正文里搜「服务方式」四字，靠硬编码「2026-05-09 前不检查」掩盖。实测样本 540 行**正文命中 0 行**，而独立的 `服务方式` 列在项目类/售后类 **100% 填写**、售前类 42/55——生效日一过，全量误判、合规率被打穿。

**重构后**：
- 判定 = **`服务方式` 列非空**（取值域：客户现场 / 远程 / 电话）。问题码 `MISS_SERVICE_MODE`。
- 适用：项目类 / 售前类 / 售后类。
- **生效日**抽为常量 `SERVICE_MODE_EFFECTIVE_DATE = "2026-05-09"`（`yitian_rules.py`）：`工作日 < 生效日` 的记录豁免此项检查（导入早期历史数据时不追溯问责）。改一处即可调整或取消。

### 6.4 工时类型一致性（仅售前类 / 售后类）

命中任一禁止词 → 问题码 `TYPE_MISMATCH`，消息含「应归属类型」。

- 售前类禁止词：正式上线、割接上线、生产上线、生产环境部署、项目验收、系统验收、初验、终验、验收报告（→项目类）；投标书、标书制作、招标文件（→业务类）
- 售后类禁止词：方案演示、产品演示、需求调研（→售前类）；实施部署、安装部署、项目实施、项目验收、系统验收、验收报告（→项目类）；投标、标书（→业务类）

### 6.5 产品类别（两级复核 + 同工单关联）

问题码 `PRODUCT_MISMATCH`。判定顺序：

1. `工作成果` 含本产品线（`产研侧产品线`）任一合法关键词 → 合格
2. **同工单关联**：同一 `工单编号` 下任一条工时记录的 `工作成果` 含本产品线关键词 → 合格
3. 不含本产品线关键词，但含**其他产品线**的典型关键词 → **触发二级复核**：按 `产研侧产品名称` 匹配二级关键词表 → 命中则合格（覆盖一级报错），未命中则判 `PRODUCT_MISMATCH`
4. 既不含本产品词、也不含其他产品词 → 无法判断，**不报错**
5. `工作成果` 含「项目管理」→ 跳过本项检查
6. 产品线为「一体化终端管理（天擎）」或「新天擎 V10」时，`工作成果` 含「天擎」→ 直接合格（避免「天擎+天眼」互相误判）

匹配**大小写不敏感**。排除词「组件」「租户」为云安全管理平台专属，出现在其他产品线不触发报错。产品线关键词表 22 条、二级产品名关键词表 2 条，逐字迁移（含原表的「更多产品关键词待补充」现状——后续维护改 `yitian_rules.py` 一处）。

### 6.6 客户名称一致性

适用：项目类 / 售前类 / 售后类。`客户` 列为空，但 `工作成果` 出现「客户」/「用户」/「甲方」/「业主」→ 问题码 `MISS_CUSTOMER`。

### 6.7 售前服务产品类别提示（仅提示，不计不合规）

`项目类型` 含「售前服务」**且** `工作类型三` ∉ {文档编写与汇报, 项目管理, 项目验收} **且** `产研侧产品线` == 「其他」→ 提示码 `HINT_PRESALE_PRODUCT`，状态 `ok=1`（合规（提示））。

### 6.8 合规判定与合规率

| 情况 | `chk` | `ok` |
|---|---|---|
| 无问题、无提示 | true | 0 = 合规 |
| 无问题、有提示 | true | 1 = 合规（提示） |
| 存在任一问题 | true | 2 = 问题 |
| 管理类 | true | 0 = 合规（跳过检查，但计入合规率分母） |
| 业务类 / 假期类 / 工时=0 | **false** | 0（不计入合规率分子分母） |

**合规率 = count(chk 且 ok∈{0,1}) ÷ count(chk)**。分母含管理类，不含业务类/假期类/0 工时——与原工具一致。

## 7. 指标口径

在**用户选定的日期区间**内计算（前端 `lib/yitian/*.ts` 纯函数，源数据 = `entries` + `days` + `roster`）：

| 指标 | 口径 |
|---|---|
| 基础工时（人均） | 区间内**工作日数 × 8h** |
| 实际工时（员工） | 该员工区间内**全部工时类型**的工时之和 |
| 饱和度 | 实际工时 ÷ 基础工时 × 100%；基础工时 = 0 → `null`（前端显 `-`） |
| 加班工时 | 实际工时 − 基础工时（> 0 计加班） |
| 未按时填写 | **有记录但**实际工时 − 基础工时 < 0（差值为负） |
| **完全未填人员** | `roster` 中区间内**零条工时记录**的人（原工具盲区） |
| KPI「未填人数」 | 未按时填写人数 **+** 完全未填人数（两个清单互斥，KPI 取合计） |
| 补全后饱和度 | 将上述两类未填人员按基础工时补齐后的平均饱和度 |
| 工时类型占比 | 各类型工时 ÷ 总工时（含管理类/业务类/假期类） |
| 分层汇总 | L3 → L3-1 → L4（**取自花名册**，非工时表列） |
| TOP1000 支持 | 按 L4 汇总：总工时、TOP1000 工时、占比、TOP1000 客户数（去重）。仅统计 项目类/售前类/售后类。**TOP1000 判定 = `客户` ∈ `input/TOP1000.xlsx` 且 `客户级别 == TOP1000大客户`（复用 `projects.read_top1000` + `config.TOP1000_LEVEL`）**；组织维度直接按花名册 L4 分组，**废弃原工具那份写死的 13 组织清单与模糊匹配** |
| 跨 BG 支持 | 仅统计 项目类/售前类。本 BG = `销售L2组织` ∈ `THIS_BG_L2_ORGS`（东北大区/京津冀大区/小金融集团军/银行集团军/运营商集团军/交付中心，常量在 `yitian_rules.py`）；其余为跨 BG。输出本/跨 BG 工时与占比 |

## 8. 后端端点与权限

| 方法 | 路径 | 鉴权 | 说明 |
|---|---|---|---|
| GET | `/api/yitian/data` | 登录 **+ 持有任一 yitian pageKey**（或超管/`*`） | 返回 `yitian_data.json`。**超管或 `allowedL4` 含 `*` → 全量；否则按 `allowedL4` 服务端切数据** |

> **纵深防御**：`/api/yitian/data` 比 `/data/analysis_data.json`（仅需登录）多加一道 pageKey 闸——工时是员工级数据，未被授予 `/yitian` 任一页面的账号，连 curl 也不该拿到自己 L4 的工时明细。判定：`allowedPages` 含 `*` 或含 `yitian` / `yitian-compliance` / `yitian-analytics` / `yitian-trend` / `yitian-customer` 任一。

**上传**：扩展现有 `/api/inputs/upload`——`config.py` 新增 `INPUT_SUBDIR_MAP = {"工时.xlsx": "yitian", "holidays.csv": "yitian"}`，白名单命中该表则写入 `input/<subdir>/`，否则维持写 `input/` 根。文件名严格白名单精确匹配（不拼接用户输入，无穿越面）。`/api/files/status` 同步纳入这两个文件的状态展示。

**L4 数据切分**（仿 `handle_data_json` + `data_scope` 成熟范式，函数落在**现有 `data_scope.py`** 内，与 `filter_analysis_data` 并列，切分逻辑单点收敛）：

```
scope_yitian_data(data, allowed_l4) →
  roster  : 只留 l4 ∈ allowed_l4 的人
  entries : 只留 e(工号) ∈ 保留 roster 的行；下标重排
  issues  : 只留 i 指向保留 entries 的行；i 重映射
  dims/days/meta : 原样（无个人信息；meta.employees 重算为可见人数）
```

**普通管理员在网络层面就拿不到他人 L4 的员工与问题明细**——不是前端隐藏。

**受保护路径**：`data/yitian_data.json` 加入 `_is_protected_data_path`（非超管直连原始文件路径 → 403），强制走 `/api/yitian/data` 的切分逻辑。

## 10. 测试策略

**pytest（`tests/test_yitian*.py`）**——先写测试再写实现（CLAUDE.md 硬性要求）：
- 工作日：正常周、含法定假、含调休上班日、`holidays.csv` 缺失降级、日期超出 csv 覆盖范围

- 工号 join：大小写归一、工时表有花名册无（离职）、花名册有工时无（完全未填）、重名不串
- 合规 6 类规则各自的命中/不命中/边界；服务方式生效日豁免；同工单关联；二级复核覆盖一级；天擎特例

- 指标：饱和度（基础工时 0 → null）、加班、未填、合规率、TOP1000、跨 BG
- 隐私：输出 JSON 不含电话/籍贯/岗位；合规行不含工作成果正文
- L4 切分：普通账号拿不到他人 L4 的 roster/entries/issues；下标重映射正确（`issues[].i` 仍指向正确的 entry）
- 管线：`input/yitian/工时.xlsx` 缺失时主管线不受影响

**vitest**：`lib/yitian/*.ts` 各口径；周口径切换的等价性（同一区间两种周划分下总量一致）。

**契约**：`schema.py` → JSON Schema → `npm run gen:types`，前后端类型同源。

**验收**：`bash verify.sh` 全绿（语法 + ruff + pytest + 前端 typecheck/vitest/build）。真实数据冒烟：用 `yitian-analyze/工时.xlsx` 导入，与原工具 HTML 报告逐项核对总工时/合规数/饱和度（**服务方式一项预期不同——那是本次修正的缺陷，须确认差异只来自它**）。
