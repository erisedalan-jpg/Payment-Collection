# 服务器部署手册（Ubuntu 全新部署）

> 目标：在一台全新 Ubuntu 服务器上从零部署「项目管理平台」精简版（LTS-1.0.0），供内网多名管理人员并发访问。
> 架构：**nginx（对外 :80，仅接管 `/pm/` 路径前缀）→ 反向代理 → app（127.0.0.1:8080，systemd 托管）**。
> 部署目录：`/opt/pmplatform`；前端用 `--base=/pm/` 构建，对外首页地址为 `http://<服务器IP或域名>/pm`。
> 适用系统：Ubuntu 22.04 / 24.04 LTS。

---

## 0. 架构说明

- 前端以 `--base=/pm/` 构建：静态资源路径为 `/pm/assets/...`，路由 base 为 `/pm/`，所有接口/数据请求都带 `/pm` 前缀（`/pm/api`、`/pm/data`）。这样即使服务器上还跑着其它系统占用根路径 `/`，本平台也不会冲突。
- nginx 只需一个 `location /pm/`，把 `/pm/xxx` 去掉前缀后转发给 app（`proxy_pass` 末尾带 `/`）。
- app（`server.py`）本身默认只绑定 `127.0.0.1:8080`（回环地址），只允许本机 nginx 访问，不对外网/局域网直接暴露。

## 1. 系统准备

```bash
sudo apt update
sudo apt install -y python3-venv nodejs npm git
# 若发行版自带 nodejs 版本过低（< 18），改用 NodeSource 源或 nvm 安装 Node 18+，此处不展开

# 专用低权用户（不可登录 shell）
sudo useradd -r -s /usr/sbin/nologin pmapp
# 固定东八区（回款延期判定依赖服务器时区）
sudo timedatectl set-timezone Asia/Shanghai

# 部署目录
sudo mkdir -p /opt/pmplatform
```

检查 Node/npm 版本：

```bash
node --version    # 期望 18+
npm --version
```

## 2. 上传 / 克隆代码

任选其一，把代码放到 `/opt/pmplatform`：

```bash
# 方式 A：git 克隆
sudo git clone <仓库地址> /opt/pmplatform

# 方式 B：本地打包上传（scp / rsync）
# 在本地把 lts/ 目录整体传到服务器 /opt/pmplatform
```

```bash
sudo chown -R pmapp:pmapp /opt/pmplatform    # data/、input/ 需 pmapp 可写
```

## 3. 后端依赖（venv）

```bash
sudo -u pmapp python3 -m venv /opt/pmplatform/.venv
sudo -u pmapp /opt/pmplatform/.venv/bin/pip install -r /opt/pmplatform/requirements.txt
sudo -u pmapp /opt/pmplatform/.venv/bin/python -c "import pydantic, openpyxl; print('deps OK')"
```

## 4. 构建前端（`/pm` 路径前缀）

```bash
cd /opt/pmplatform/frontend
sudo -u pmapp npm ci
sudo -u pmapp npx vite build --base=/pm/
```

> 服务器上直接用 bash 执行即可，`--base=/pm/` 不会被路径篡改（该问题仅出现在 Windows 上用某些 shell 打包时，Linux 环境不受影响）。

校验构建产物已带 `/pm` 前缀：

```bash
grep -o '/pm/assets[^"]*' /opt/pmplatform/frontend/dist/index.html | head -1   # 应有输出
```

> 可选：仓库自带 `make_deploy_zip.py` 可在本地一次性打包「代码 + 已构建 dist + 数据」为部署压缩包，适合离线环境整体搬运；本手册按在线构建的标准路径说明，两种方式二选一即可。

## 5. systemd 托管

新建 `/etc/systemd/system/pmplatform.service`：

```ini
[Unit]
Description=Project Management Platform (LTS)
After=network.target

[Service]
Type=simple
User=pmapp
Group=pmapp
WorkingDirectory=/opt/pmplatform
Environment=TZ=Asia/Shanghai
ExecStart=/opt/pmplatform/.venv/bin/python server.py
Restart=always
RestartSec=3
NoNewPrivileges=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now pmplatform
sudo systemctl status pmplatform    # 期望 active (running)
```

> 生命周期交给 systemd 管理：停止/重启用 `sudo systemctl stop|restart pmplatform`，不要用 `python server.py --stop`（依赖的进程查找方式在部分 Linux 环境下可能不可靠）。

## 6. nginx 反向代理

新建（或复用已有 :80 server 块，追加下列 `location`）`/etc/nginx/sites-available/pmplatform`：

```nginx
server {
    listen 80;
    server_name pm.intranet.example;    # 改成本服务器实际域名/IP

    client_max_body_size 100M;          # 允许上传 PMIS/CSV 大文件
    gzip on;
    gzip_types application/json application/javascript text/css text/plain;
    gzip_min_length 1024;

    location = /pm { return 301 /pm/; }
    location /pm/ {
        proxy_pass http://127.0.0.1:8080/;      # 末尾 / 关键：把 /pm/xxx 重写为 /xxx 再转给 app
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;                     # 「更新数据」SSE 进度流需要实时透传
        proxy_read_timeout 3600s;                # 大数据量处理耗时较长
    }
}
```

```bash
sudo ln -s /etc/nginx/sites-available/pmplatform /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

若服务器与其它系统共享 :80 且已有 server 块占用根路径，可只把上面的两段 `location` 粘进现有 server 块，不新建独立 server（避免与其它系统的 catch-all 冲突）。

防火墙放行：

```bash
sudo ufw allow 80/tcp
# 或按内网网段限定：sudo ufw allow from 10.0.0.0/8 to any port 80
```

## 7. 首次放数据 + 生成分析数据

把数据源文件放到 `/opt/pmplatform/input/`：PMIS 九表放 `input/pmis/`，`collection_stages.csv`、`payment_records.csv`、`profit_loss_direct.csv`、`profit_loss_bridge.csv`、预算科目对照数据 CSV、`delivery_analysis.xlsx`、`组织架构.xlsx`、`A.xlsx`、`TOP1000.xlsx` 放 `input/`（完整文件名清单以登录后「数据管理」页面「文件状态」区展示为准）。

```bash
sudo chown -R pmapp:pmapp /opt/pmplatform/input
```

浏览器打开 `http://<服务器IP或域名>/pm`，登录后进入「数据管理」（`/data`）页面，确认文件已识别（或直接用页面上传功能补齐），点击「更新数据」，等待处理完成生成 `data/analysis_data.json`。

也可以跳过页面操作，直接在服务器上手工触发一次：

```bash
sudo -u pmapp /opt/pmplatform/.venv/bin/python /opt/pmplatform/preprocess_data.py
```

## 8. 建超管账号 / 改默认口令

首次启动时，服务会按源码内置的种子超管配置自动建号（见 `auth.py`），用于首次登录。**上线前必须逐个改掉种子超管的默认口令**（超管口令无法在「账号管理」页面里自行修改，需用助手脚本）：

```bash
sudo -u pmapp /opt/pmplatform/.venv/bin/python /opt/pmplatform/reset_super_password.py <账号>
# 不传新口令则交互式输入（不回显），推荐用此方式，避免口令出现在 shell 历史中
```

登录后可在「账号管理」页新建其他管理员账号，按部门（L4）与可访问页面范围授权。

## 运维

**更新数据**：把新文件放到 `input/`（PMIS 九表放 `input/pmis/`），登录页面「数据管理 → 更新数据」，或服务器上直接执行：

```bash
sudo -u pmapp /opt/pmplatform/.venv/bin/python /opt/pmplatform/preprocess_data.py
```

**备份 `data/` 目录**：账号、跟进记录、项目标签、门户配置、主数据、历史快照均在 `data/` 下，建议每日异机备份：

```bash
0 2 * * * tar czf /backup/pmplatform-$(date +\%F).tgz -C /opt/pmplatform data/
```

重点文件：`data/accounts.json`（口令哈希）、`data/analysis_data.json`、`data/followup_records.json`、`data/project_tags.json`、`data/history/`。

**查看日志**：

```bash
journalctl -u pmplatform -f
# 或应用自身日志文件：/opt/pmplatform/log/server.log
```

**重启服务**：

```bash
sudo systemctl restart pmplatform
```

**回滚**：
- 代码回滚：`git checkout <上一个版本 tag/commit>` 后重新构建前端、`systemctl restart pmplatform`。
- 数据回滚：应用内置「数据管理 → 数据历史」保留最近若干次「更新数据」快照，可在页面内一键回滚；更早的数据以第 8 步的 `data/` 备份包恢复。

## 安全提示

- 本平台面向**可信内网**设计，不建议直接暴露在公网。
- app 默认只绑定 `127.0.0.1:8080` 回环地址，仅接受本机 nginx 的转发请求；部署时不要为图方便把监听地址改成对外网卡或跳过 nginx 直接开放 8080 端口。
- 账号权限按部门（L4）维度隔离：非超管账号只能看到被授权 L4 范围内的数据，写操作与运维类接口（更新数据、清空数据等）默认限超管。
- 端口 80 为明文 HTTP，会话与数据未加密，仅限可信内网网段访问；如需面向不可信网络开放，建议在 nginx 层加 TLS（证书配置不改变本手册的 `/pm` 路径结构）。
- 定期按上文「运维」一节备份 `data/` 目录，避免账号与业务数据丢失后无法恢复。
